## 1. What is Sorting?

Sorting is the process of arranging elements in a **specific order** (ascending, descending, or custom). It is one of the most fundamental operations in computer science — sorting unlocks efficient searching, grouping, and many other operations.

### Why Sorting Matters

- **Enables Binary Search** — O(log n) instead of O(n).
- **Simplifies problems** — duplicates are adjacent, pairs are easier to find.
- **Required for many algorithms** — merge intervals, two pointers, greedy.
- **Database indexing** — B-Trees keep data sorted for fast queries.
- **Real-world ubiquity** — leaderboards, feeds, search results, file systems.

### Key Insight

> Sorting often turns an O(n²) brute-force problem into an O(n log n) + O(n) solution. **When stuck on an interview problem, ask: "What if I sort it first?"**

### By Approach

## 3. Comparison of All Sorting Algorithms

| Algorithm | Best | Average | Worst | Space | Stable? | Use When |
|-----------|------|---------|-------|-------|---------|----------|
| **Bubble Sort** | O(n) | O(n²) | O(n²) | O(1) | ✅ | Never in production; teaching only |
| **Selection Sort** | O(n²) | O(n²) | O(n²) | O(1) | ❌ | Minimizing swaps (write-expensive memory) |
| **Insertion Sort** | O(n) | O(n²) | O(n²) | O(1) | ✅ | Small n, nearly sorted data, online sorting |
| **Merge Sort** | O(n log n) | O(n log n) | O(n log n) | O(n) | ✅ | Guaranteed O(n log n), stability needed, linked lists |
| **Quick Sort** | O(n log n) | O(n log n) | O(n²) | O(log n) | ❌ | General purpose, fastest in practice |
| **Heap Sort** | O(n log n) | O(n log n) | O(n log n) | O(1) | ❌ | Guaranteed O(n log n) + in-place |
| **Counting Sort** | O(n + k) | O(n + k) | O(n + k) | O(k) | ✅ | Small range integers |
| **Radix Sort** | O(d(n+k)) | O(d(n+k)) | O(d(n+k)) | O(n+k) | ✅ | Fixed-length integers/strings |

> **Lower bound for comparison sorts: Ω(n log n).** No comparison-based sort can do better. Non-comparison sorts bypass this by not comparing elements.

### 4.1 Bubble Sort

#### 🧠 Intuition — "Bubbles Rise"

Imagine the array is a **vertical column of water with numbered bubbles**. Heavier (larger) bubbles **sink to the bottom**, and lighter (smaller) ones **float to the top**. In each pass, you walk through the array comparing adjacent elements and **swapping** them if they're out of order. After the first pass, the **largest element "bubbles up"** to the last position. After the second pass, the second-largest is in place. And so on.

> **Think of it as:** "In each pass, the biggest unsorted element floats to its correct position at the end."

```
Pass 1: [5, 3, 8, 1, 2]
         3  5           → swap(5,3)
            5  8        → no swap
               8  1     → swap(8,1)
                  8  2  → swap(8,2)
         [3, 5, 1, 2, 8]  ← 8 is now in place ✅

Pass 2: [3, 5, 1, 2, | 8]
         3  5        → no swap
            5  1     → swap
               5  2  → swap
         [3, 1, 2, 5, 8]  ← 5 is now in place ✅

... and so on
```

#### Code

```cpp
void bubbleSort(vector<int>& arr) {
    int n = arr.size();
    for (int i = 0; i < n - 1; i++) {
        bool swapped = false;
        for (int j = 0; j < n - 1 - i; j++) {
            if (arr[j] > arr[j + 1]) {
                swap(arr[j], arr[j + 1]);
                swapped = true;
            }
        }
        if (!swapped) break; // already sorted — adaptive!
    }
}
```

#### Analysis

- **Best:** O(n) — when already sorted (with `swapped` optimization).
- **Worst:** O(n²) — reverse sorted. Every pair requires a swap.
- **Swaps:** O(n²) in worst case — very expensive.
- **When to use:** Never in practice. Only for teaching. But know it for interviews.

### 4.2 Selection Sort

#### 🧠 Intuition — "Find the Minimum and Place It"

Imagine you have a **deck of unsorted cards** laid out on a table. You scan through ALL the cards to find the **smallest one**, then place it in position 1. Then scan the remaining cards for the next smallest, place it in position 2. And so on.

> **Think of it as:** "Select the minimum from the unsorted part, swap it to the front of the unsorted part."

You're essentially **selecting** the correct element for each position.

```
[64, 25, 12, 22, 11]
 Find min in [0..4] → 11 (index 4). Swap with index 0.
[11, 25, 12, 22, 64]
 Find min in [1..4] → 12 (index 2). Swap with index 1.
[11, 12, 25, 22, 64]
 Find min in [2..4] → 22 (index 3). Swap with index 2.
[11, 12, 22, 25, 64]
 Find min in [3..4] → 25 (index 3). Already in place.
[11, 12, 22, 25, 64] ✅
```

#### Code

```cpp
void selectionSort(vector<int>& arr) {
    int n = arr.size();
    for (int i = 0; i < n - 1; i++) {
        int minIdx = i;
        for (int j = i + 1; j < n; j++) {
            if (arr[j] < arr[minIdx])
                minIdx = j;
        }
        swap(arr[i], arr[minIdx]);
    }
}
```

#### Analysis

- **Always O(n²)** — doesn't adapt to sorted input. Always scans the full unsorted portion.
- **Swaps:** Exactly O(n) — **minimum swaps** among all O(n²) sorts.
- **When to use:** When **memory writes are expensive** (e.g., flash memory, EEPROM). Each element is swapped at most once.
- **Not stable:** Swapping can change the order of equal elements.

#### Bubble Sort vs Selection Sort

| Aspect | Bubble Sort | Selection Sort |
|--------|-------------|----------------|
| Comparisons | O(n²) | O(n²) |
| Swaps (worst) | O(n²) | O(n) ✅ |
| Adaptive? | ✅ Yes (early termination) | ❌ No |
| Stable? | ✅ Yes | ❌ No |

### 4.3 Insertion Sort

#### 🧠 Intuition — "Sort Like You Sort Playing Cards"

Hold a hand of cards. Pick up one card at a time and **insert it into the correct position** among the cards you already hold. The cards in your hand are always sorted.

> **Think of it as:** "Take the next unsorted element and slide it into its correct position in the already-sorted part."

The "sorted part" grows from left to right. For each new element, you **shift** larger elements one step right to make room.

```
[5, 3, 8, 1, 2]
 Sorted: [5]. Take 3. 3 < 5, shift 5 right. Insert 3.
[3, 5, 8, 1, 2]
 Sorted: [3, 5]. Take 8. 8 > 5, stays. Insert 8.
[3, 5, 8, 1, 2]
 Sorted: [3, 5, 8]. Take 1. Shift 8, 5, 3 right. Insert 1.
[1, 3, 5, 8, 2]
 Sorted: [1, 3, 5, 8]. Take 2. Shift 8, 5, 3 right. Insert 2.
[1, 2, 3, 5, 8] ✅
```

#### Code

```cpp
void insertionSort(vector<int>& arr) {
    int n = arr.size();
    for (int i = 1; i < n; i++) {
        int key = arr[i];
        int j = i - 1;
        while (j >= 0 && arr[j] > key) {
            arr[j + 1] = arr[j];  // shift right
            j--;
        }
        arr[j + 1] = key;  // insert
    }
}
```

#### Analysis

- **Best:** O(n) — already sorted. Inner loop never executes.
- **Worst:** O(n²) — reverse sorted.
- **Adaptive:** Very fast on **nearly sorted** data. Each element moves at most `k` positions → O(nk).
- **Online:** Can sort data as it arrives (don't need the full array upfront).
- **When to use:**
  - Small arrays (n < 20–50). Often used as a base case in Merge/Quick Sort.
  - Nearly sorted data (best practical choice).
  - Online sorting (data arriving in a stream).

#### Why std::sort Uses Insertion Sort for Small Arrays

Quick Sort has overhead (recursion, pivot selection). For tiny arrays (< 16 elements), **Insertion Sort is faster** due to low overhead, great cache locality, and adaptiveness. Most standard library sorts (including C++ `std::sort`) switch to Insertion Sort when the partition size drops below a threshold.

### 4.4 Merge Sort

#### 🧠 Intuition — "Divide, Sort Halves, Merge"

Imagine sorting a **huge deck of exam papers**. You can't sort it all at once. So:
1. **Split** the deck into two halves.
2. Give each half to a helper. They **recursively** split and sort their halves.
3. Once both halves are sorted, **merge** them back together by comparing the top card of each pile and taking the smaller one.

> **Think of it as:** "I can't sort the whole thing, but if someone gives me two sorted halves, I can merge them in O(n). So I'll keep splitting until each piece has 1 element (trivially sorted), then merge my way back up."

The **key operation is the merge**: two sorted arrays → one sorted array in O(n) using two pointers.

```
Split phase:
  [38, 27, 43, 3, 9, 82, 10]
  [38, 27, 43]          [3, 9, 82, 10]
  [38] [27, 43]          [3, 9] [82, 10]
  [38] [27] [43]         [3] [9] [82] [10]

Merge phase (bottom-up):
  [38] [27] → [27, 38]     [43]         [3] [9] → [3, 9]     [82] [10] → [10, 82]
  [27, 38] [43] → [27, 38, 43]          [3, 9] [10, 82] → [3, 9, 10, 82]
  
  [27, 38, 43] + [3, 9, 10, 82] → [3, 9, 10, 27, 38, 43, 82] ✅
```

#### Code

```cpp
void merge(vector<int>& arr, int left, int mid, int right) {
    vector<int> temp;
    int i = left, j = mid + 1;
    while (i <= mid && j <= right) {
        if (arr[i] <= arr[j])  // <= for stability
            temp.push_back(arr[i++]);
        else
            temp.push_back(arr[j++]);
    }
    while (i <= mid) temp.push_back(arr[i++]);
    while (j <= right) temp.push_back(arr[j++]);
    for (int k = left; k <= right; k++)
        arr[k] = temp[k - left];
}

void mergeSort(vector<int>& arr, int left, int right) {
    if (left >= right) return;
    int mid = left + (right - left) / 2;
    mergeSort(arr, left, mid);
    mergeSort(arr, mid + 1, right);
    merge(arr, left, mid, right);
}
```

#### Analysis

- **Always O(n log n)** — guaranteed. No worst case degradation.
- **Space: O(n)** — needs extra array for merging. This is the main disadvantage.
- **Stable:** Equal elements maintain order (critical for multi-key sorts).
- **Best for linked lists:** Can merge in O(1) space on linked lists (just re-point pointers).
- **Parallelizable:** Each half can be sorted independently → great for multi-threaded sorting.

#### The Merge Step — Why It's O(n)

Two pointers, one on each sorted half. Compare, take the smaller, advance that pointer. Each element is touched exactly once → O(n). This O(n) merge happens at each of log n levels → total = O(n log n).

```
Level 0 (top):    merge n elements              → O(n) work
Level 1:          merge n/2 + n/2 elements       → O(n) work
Level 2:          merge n/4×4 elements            → O(n) work
...
Level log n:      merge 1×n elements              → O(n) work
Total: O(n) × O(log n) = O(n log n)
```

### 4.5 Quick Sort

#### 🧠 Intuition — "Pick a Pivot, Partition Around It"

Pick any element as a **pivot**. Rearrange the array so that:
- All elements **< pivot** go to the **left**.
- All elements **> pivot** go to the **right**.
- The **pivot is now in its correct final position**.

Then recursively sort the left and right parts.

> **Think of it as:** "I don't know where everything goes, but I CAN put ONE element in its correct position and separate everything into 'smaller' and 'larger'. Then I repeat for each half."

The **key operation is the partition**: rearrange around pivot in O(n).

```
[10, 7, 8, 9, 1, 5]   pivot = 5 (last element)

Partition: move everything < 5 to the left
  i=-1
  j=0: 10 > 5 → skip
  j=1: 7 > 5  → skip
  j=2: 8 > 5  → skip
  j=3: 9 > 5  → skip
  j=4: 1 < 5  → i=0, swap arr[0] and arr[4] → [1, 7, 8, 9, 10, 5]
  Place pivot: swap arr[1] and arr[5] → [1, 5, 8, 9, 10, 7]
                                            ↑
                                        5 is in correct position!

Now sort [1] and [8, 9, 10, 7] recursively.
```

#### Code

```cpp
int partition(vector<int>& arr, int low, int high) {
    int pivot = arr[high];  // last element as pivot
    int i = low - 1;        // boundary of "smaller" section
    for (int j = low; j < high; j++) {
        if (arr[j] < pivot) {
            i++;
            swap(arr[i], arr[j]);
        }
    }
    swap(arr[i + 1], arr[high]);  // place pivot in correct position
    return i + 1;
}

void quickSort(vector<int>& arr, int low, int high) {
    if (low < high) {
        int pivotIdx = partition(arr, low, high);
        quickSort(arr, low, pivotIdx - 1);
        quickSort(arr, pivotIdx + 1, high);
    }
}
```

#### Analysis

- **Average: O(n log n)** — when pivot splits roughly in half.
- **Worst: O(n²)** — when pivot is always min or max (already sorted + last-element pivot).
- **Space: O(log n)** for recursion stack — in-place (no extra array like merge sort).
- **Fastest in practice** — excellent cache locality, small constant factors.

#### Pivot Selection Strategies

| Strategy | Risk | Recommendation |
|----------|------|---------------|
| First/Last element | O(n²) on sorted input | ❌ Avoid |
| Random element | Expected O(n log n) | ✅ Good |
| Median of three | Very unlikely O(n²) | ✅✅ Best for practice |
| Median of medians | Guaranteed O(n log n) | Overkill for most uses |

```cpp
// Randomized pivot (best practical choice)
int randomPartition(vector<int>& arr, int low, int high) {
    int random = low + rand() % (high - low + 1);
    swap(arr[random], arr[high]);
    return partition(arr, low, high);
}
```

#### Quick Sort vs Merge Sort — The Classic Interview Question

| Aspect | Quick Sort | Merge Sort |
|--------|-----------|------------|
| Average time | O(n log n) ✅ | O(n log n) |
| Worst time | O(n²) ❌ | O(n log n) ✅ |
| Space | O(log n) ✅ | O(n) |
| Stable? | ❌ | ✅ |
| Cache performance | Excellent ✅ (in-place) | Poor (extra array) |
| Practical speed | Faster ✅ | Slower |
| Best for | Arrays | Linked lists |
| Used in std::sort? | ✅ (Introsort = Quick + Heap + Insertion) | Used in stable_sort |

### 4.6 Heap Sort

#### 🧠 Intuition — "Build a Max-Heap, Extract Maximum Repeatedly"

Think of a **tournament bracket**. In a max-heap, the largest element is always at the root (the champion). 

1. **Build a max-heap** from the array — the largest element is now at index 0.
2. **Swap** the root (largest) with the last element → largest is now at the end (correct position).
3. **Shrink** the heap by 1 and **heapify** the root down to restore the heap property.
4. Repeat until the heap is empty.

> **Think of it as:** "The heap always tells me the maximum for free. I extract it, put it at the end, fix the heap, and repeat."

```
Array: [4, 10, 3, 5, 1]

Step 1 — Build Max-Heap:
         10
        /  \
       5    3
      / \
     4   1
  Array: [10, 5, 3, 4, 1]

Step 2 — Extract 10, swap with last:
  [1, 5, 3, 4, | 10]   Heapify down:
         5
        / \
       4   3
      /
     1
  Array: [5, 4, 3, 1, | 10]

Step 3 — Extract 5:
  [1, 4, 3, | 5, 10]   Heapify:
  [4, 1, 3, | 5, 10]

... continue until fully sorted.
  [1, 3, 4, 5, 10] ✅
```

#### Code

```cpp
void heapify(vector<int>& arr, int n, int i) {
    int largest = i;
    int left = 2 * i + 1;
    int right = 2 * i + 2;
    if (left < n && arr[left] > arr[largest]) largest = left;
    if (right < n && arr[right] > arr[largest]) largest = right;
    if (largest != i) {
        swap(arr[i], arr[largest]);
        heapify(arr, n, largest);
    }
}

void heapSort(vector<int>& arr) {
    int n = arr.size();
    // Build max-heap (start from last non-leaf node)
    for (int i = n / 2 - 1; i >= 0; i--)
        heapify(arr, n, i);
    // Extract elements one by one
    for (int i = n - 1; i > 0; i--) {
        swap(arr[0], arr[i]);
        heapify(arr, i, 0);
    }
}
```

#### Analysis

- **Always O(n log n)** — guaranteed, no worst case.
- **Space: O(1)** — fully in-place (the best of both worlds: guaranteed time + in-place).
- **Not stable:** Heapify swaps distant elements.
- **Poor cache performance:** Accesses `2i+1`, `2i+2` → jumps around in memory.
- **When to use:** When you need guaranteed O(n log n) AND can't afford O(n) extra space.

#### Building a Heap — Why O(n) and Not O(n log n)?

You might think building a heap is O(n log n) (inserting n elements, each O(log n)). But building **bottom-up** is O(n):

- Leaves (n/2 nodes): 0 work.
- Level above: n/4 nodes × 1 step.
- Next: n/8 nodes × 2 steps. ... Total = n × (1/4 + 2/8 + 3/16 + ...) = O(n).

### 4.7 Tim Sort (What Your Language Actually Uses)

#### 🧠 Intuition — "Real-world data has patterns. Exploit them."

Tim Sort (used by Python's `sort()`, Java's `Arrays.sort()`, and often in C++ `stable_sort`) is a **hybrid** of Merge Sort and Insertion Sort, designed for real-world data.

1. Divide the array into small "runs" (naturally sorted subsequences or forced via Insertion Sort for chunks < 32–64).
2. Merge runs using an optimized merge that exploits existing order.

> **Think of it as:** "Most real data is partially sorted. Find the sorted parts, then merge them."

- **Time:** O(n) best case (already sorted), O(n log n) worst case.
- **Space:** O(n).
- **Stable:** ✅ Yes.

## 5. Non-Comparison Based Sorts

These break the O(n log n) barrier by **not comparing elements** — they use element values as indices.

### 5.1 Counting Sort

#### 🧠 Intuition — "Count, Then Place"

If you know the range of values (0 to k), **count how many times each value appears**, then use the counts to place elements directly.

> **Think of it as:** "I have 3 zeros, 2 ones, 5 twos → output: 000 11 22222"

```
Input:  [4, 2, 2, 8, 3, 3, 1]
Range:  0–8

Count array:
  index: 0  1  2  3  4  5  6  7  8
  count: 0  1  2  2  1  0  0  0  1

Output: [1, 2, 2, 3, 3, 4, 8]
```

```cpp
void countingSort(vector<int>& arr, int maxVal) {
    vector<int> count(maxVal + 1, 0);
    for (int x : arr) count[x]++;
    int idx = 0;
    for (int i = 0; i <= maxVal; i++)
        while (count[i]-- > 0)
            arr[idx++] = i;
}
```

- **Time:** O(n + k), where k = range of values.
- **Space:** O(k).
- **When to use:** Small range of integers (e.g., ages 0–150, grades 0–100, characters a–z).

---

### 5.2 Radix Sort

#### 🧠 Intuition — "Sort Digit by Digit"

Sort numbers by their **least significant digit first**, then second digit, etc. Each digit-sort is done using a **stable** sort (like Counting Sort).

> **Think of it as:** "Sort by ones place → sort by tens place → sort by hundreds place. After all digits, the array is fully sorted."

Why does this work? Because each digit-level sort is **stable**, so the previous digit's ordering is preserved.

```
Input: [170, 45, 75, 90, 802, 24, 2, 66]

Sort by ones digit:  [170, 90, 802, 2, 24, 45, 75, 66]
Sort by tens digit:  [802, 2, 24, 45, 66, 170, 75, 90]
Sort by hundreds:    [2, 24, 45, 66, 75, 90, 170, 802] ✅
```

- **Time:** O(d × (n + k)), d = number of digits.
- **Space:** O(n + k).
- **When to use:** Large n, fixed-length keys (phone numbers, IDs, IP addresses).

---

### 5.3 Bucket Sort

#### 🧠 Intuition — "Scatter into Buckets, Sort Each Bucket"

Divide the value range into **equal-width buckets**. Scatter elements into buckets based on value. Sort each bucket individually (with Insertion Sort). Concatenate.

> **Think of it as:** "Roughly distribute elements into bins, then fine-tune each bin."

- **Time:** O(n + k) average when data is **uniformly distributed**.
- **Worst:** O(n²) — all elements in one bucket.
- **When to use:** Floating-point numbers uniformly distributed in [0, 1). Or any uniformly distributed data.

## 6. C++ STL Sorting

### std::sort (Introsort)

```cpp
#include <algorithm>

vector<int> v = {5, 3, 8, 1, 2};
sort(v.begin(), v.end());                    // ascending
sort(v.begin(), v.end(), greater<int>());    // descending

// Custom comparator
sort(v.begin(), v.end(), [](int a, int b) {
    return a > b;  // descending
});

// Sort by custom property
vector<pair<int,int>> points = {{3,4}, {1,2}, {5,0}};
sort(points.begin(), points.end(), [](auto& a, auto& b) {
    return a.second < b.second;  // sort by y-coordinate
});
```

### What std::sort Actually Does (Introsort)

`std::sort` is **Introsort** = Quick Sort + Heap Sort + Insertion Sort:
1. Start with **Quick Sort** (fast average case).
2. If recursion depth exceeds 2 × log₂(n), switch to **Heap Sort** (prevents O(n²) worst case).
3. When partition size < 16, switch to **Insertion Sort** (fast for small arrays).

Result: **O(n log n) guaranteed**, fastest in practice, but **not stable**.

### std::stable_sort (Merge Sort)

```cpp
stable_sort(v.begin(), v.end());  // O(n log n), stable, O(n) extra space
```

Use when relative order of equal elements must be preserved (e.g., sort by score, then by name).

### Partial Sort & Nth Element

```cpp
// Put the 3 smallest elements at the front (sorted)
partial_sort(v.begin(), v.begin() + 3, v.end());

// Put the kth element in its correct position (unsorted rest)
nth_element(v.begin(), v.begin() + k, v.end());  // O(n) average!
```

## 7. Pattern Recognition (10 Patterns)

### Pattern 1: Sort + Two Pointers

**What:** Sort first, then use two pointers to find pairs/triplets.  
**When:** Sum problems (Two Sum on sorted, 3Sum, 4Sum).

```cpp
// 3Sum: find all triplets summing to 0
sort(arr.begin(), arr.end());
for (int i = 0; i < n - 2; i++) {
    if (i > 0 && arr[i] == arr[i-1]) continue; // skip duplicates
    int left = i + 1, right = n - 1;
    while (left < right) {
        int sum = arr[i] + arr[left] + arr[right];
        if (sum == 0) { /* found */ left++; right--; }
        else if (sum < 0) left++;
        else right--;
    }
}
```

---

### Pattern 2: Sort + Binary Search

**What:** Sort the array, then use binary search for lookups.  
**When:** Checking existence or finding closest element after sorting.

**Problems:** Two Sum (sort + BS), 4Sum II, Search in sorted matrix.

---

### Pattern 3: Merge Sort for Counting

**What:** Modify merge sort's merge step to count inversions, smaller elements after self, etc.  
**When:** Problems that need O(n log n) counting of ordered relationships.

**Problems:** Count Inversions, Count Smaller Numbers After Self, Reverse Pairs.

---

### Pattern 4: Interval Sorting

**What:** Sort intervals by start time (or end time), then process.  
**When:** Merging, scheduling, overlap detection.

**Problems:** Merge Intervals, Non-Overlapping Intervals, Meeting Rooms I & II, Minimum Platforms.

---

### Pattern 5: Custom Comparator

**What:** Define a custom ordering for sorting.  
**When:** Sort by custom criteria, lexicographic tricks.

```cpp
// Largest Number: arrange numbers to form the largest string
sort(nums.begin(), nums.end(), [](string& a, string& b) {
    return a + b > b + a;  // "30" + "3" = "303" vs "3" + "30" = "330"
});
```

**Problems:** Largest Number, Custom Sort String, Relative Sort Array.

---

### Pattern 6: Dutch National Flag (3-Way Partition)

**What:** Partition into 3 groups in a single pass.  
**When:** Sort {0,1,2}, partition around a value.

**Problems:** Sort Colors, Segregate 0s 1s 2s, Three-Way Partition.

---

### Pattern 7: Sorting as Preprocessing for Greedy

**What:** Sort to enable greedy choices (smallest first, earliest deadline, etc.).  
**When:** Assignment, scheduling, overlaps.

**Problems:** Assign Cookies, Minimum Number of Arrows, Activity Selection.

---

### Pattern 8: Topological Sort (Graph Ordering)

**What:** Order nodes so that every directed edge u → v has u before v.  
**When:** Task scheduling, build systems, course ordering.

**Problems:** Course Schedule I & II, Alien Dictionary, Task Scheduler.

---

### Pattern 9: Counting Sort / Bucket Sort as Optimization

**What:** When range is small, use non-comparison sort for O(n).  
**When:** Frequencies, small integer ranges, radix-like problems.

**Problems:** Sort Colors, Maximum Gap, Top K Frequent Elements (bucket sort).

---

### Pattern 10: Quick Select (Kth Element)

**What:** Find the kth smallest/largest without fully sorting. Uses partition from Quick Sort.  
**When:** Kth largest, median, top K elements.

```cpp
int quickSelect(vector<int>& arr, int low, int high, int k) {
    int pivotIdx = partition(arr, low, high);
    if (pivotIdx == k) return arr[k];
    else if (pivotIdx < k) return quickSelect(arr, pivotIdx + 1, high, k);
    else return quickSelect(arr, low, pivotIdx - 1, k);
}
```

- **Average: O(n)**, Worst: O(n²) — use randomized pivot.
- This is how `std::nth_element` works.

**Problems:** Kth Largest Element, Top K Frequent, Find Median from Data Stream.

## 9. Common Mistakes & Edge Cases

### ❌ Common Mistakes

| Mistake | Why it's wrong | Fix |
|---------|---------------|-----|
| Using wrong comparator | Undefined behavior in sort if not strict weak ordering | Must be irreflexive: `comp(a, a) = false` |
| Not handling **duplicates** | Extra entries in two-pointer approaches | Skip duplicates with `while (arr[i] == arr[i-1]) i++` |
| Choosing bad pivot in Quick Sort | O(n²) on sorted arrays | Use random pivot or median-of-three |
| Using unstable sort when stability needed | Relative order of equal elements changes | Use `stable_sort` |
| Sorting when not needed | O(n log n) overhead when O(n) solution exists | Consider hash map or counting |
| Modifying comparator mid-sort | Crashes or wrong results | Comparator must be consistent |

### 🧪 Edge Cases to Always Test

- **Empty array** `[]`
- **Single element** `[1]`
- **Already sorted** (ascending and descending)
- **All elements identical** `[5, 5, 5, 5]`
- **Two elements** `[2, 1]`
- **Negative numbers** `[-3, -1, -5]`
- **Very large arrays** (performance testing)
- **Array with duplicates** (stability matters)

## 10. Interview Tips & Frequently Asked Concepts

### 💡 Top Interview Tips

1. **"Why not just sort it?"** — always consider if sorting simplifies the problem. O(n log n) is often good enough.

2. **Know the trade-offs** between Quick, Merge, and Heap sort cold.

3. **Know when NOT to sort:** If you need O(n), sorting adds O(n log n) overhead. Use hash map or counting instead.

4. **Stability matters:** When asked to sort by multiple keys, sort by the secondary key first, then stable-sort by primary key.

5. **Quick Select** is O(n) average for kth element — don't sort the full array if you only need kth element.

### 🎯 Frequently Asked Interview Concepts

#### Q1: Why is the lower bound for comparison sorts O(n log n)?

**A:** Any comparison sort builds a **decision tree** of comparisons. There are n! possible orderings. A binary tree with n! leaves has height at least log₂(n!) = Ω(n log n) (by Stirling's approximation). So at least Ω(n log n) comparisons are needed.

#### Q2: Quick Sort vs Merge Sort — which would you use?

**A:**
- **Quick Sort** for arrays when average speed matters (cache-friendly, in-place).
- **Merge Sort** for linked lists (no random access needed, O(1) space merge), when stability is needed, or when guaranteed O(n log n) is required.

#### Q3: What is the best sorting algorithm?

**A:** There is no single best. It depends on:
- **Small n (< 50):** Insertion Sort.
- **General purpose:** Quick Sort (Introsort in practice).
- **Stability needed:** Merge Sort.
- **In-place + guaranteed:** Heap Sort.
- **Small integer range:** Counting Sort.
- **Nearly sorted:** Insertion Sort or Tim Sort.

#### Q4: What sort does std::sort use?

**A:** **Introsort** = Quick Sort (fast average) + Heap Sort (prevents O(n²) worst case) + Insertion Sort (fast for small partitions). Guaranteed O(n log n), not stable.

#### Q5: What is stability in sorting and why does it matter?

**A:** Stable sort preserves the relative order of equal elements. Example: sorting students by grade — if Alice and Bob both have grade A, a stable sort keeps them in their original order. Needed for multi-key sorting: sort by last name, then stable-sort by first name.

#### Q6: How would you sort 1 million 32-bit integers?

**A:** `std::sort` (Introsort) → O(n log n) ≈ 20 million operations. For tighter performance: Radix Sort on 4 bytes → O(4 × n) = O(4 million operations). Radix Sort wins for large n with bounded key sizes.

#### Q7: How would you sort 1 TB of data that doesn't fit in memory?

**A:** **External Merge Sort:**
1. Read chunks that fit in memory (e.g., 1 GB).
2. Sort each chunk in-memory.
3. Write sorted chunks to disk.
4. **K-way merge** all sorted chunks using a min-heap.
This is how databases sort large datasets.

#### Q8: What is the difference between internal and external sorting?

**A:**
- **Internal:** All data fits in RAM. Quick Sort, Merge Sort, etc.
- **External:** Data too large for RAM. External Merge Sort using disk.

#### Q9: Can you sort in O(n) time?

**A:** Yes, if elements have bounded range. Counting Sort O(n + k), Radix Sort O(d × n), Bucket Sort O(n) average. These are **not comparison-based** so they bypass the Ω(n log n) lower bound.

#### Q10: What is inversion count and how to compute it?

**A:** An inversion is a pair (i, j) where i < j but arr[i] > arr[j]. It measures how far the array is from sorted. Compute in O(n log n) by counting swaps during **merge sort** — every time an element from the right half is merged before an element from the left, count the inversions.

## 11. Sorting in Software Development & Web Development — Real-World Scenarios

> In SDE/web-dev interviews, sorting comes up in **system design, API design, and product scenarios**. This section covers real-world situations where sorting knowledge impresses interviewers.

---

### 🔍 Scenario 1: Search Results Ranking

**Interviewer:** _"Google returns millions of results in < 1 second. How are they sorted? What algorithm?"_

**Answer:**
- Search engines compute a **relevance score** for each result (PageRank + text matching + freshness + user signals).
- Results are sorted by this score using **partial sort** (only need top 10-20 results).
- Use a **min-heap of size K** → O(n log K) instead of O(n log n) full sort.
- For pagination: maintain a sorted buffer, use **merge** to combine results from distributed shards.

**Complexity:** O(n log K) where K = page size (10-20). Much faster than O(n log n).

---

### 🔍 Scenario 2: Feed Algorithms — Social Media Timeline

**Interviewer:** _"Instagram shows posts in a 'best posts first' order. How does the feed system work?"_

**Answer:**
- Each post gets a score: `relevance = f(engagement, recency, relationship, content_type)`.
- **Candidate generation:** Fetch ~500 candidate posts.
- **Ranking:** Sort by score using **Quick Sort** (fast, in-memory).
- **Stability matters:** If two posts have the same score, maintain chronological order → use **stable sort**.
- **Personalized re-ranking** may use ML models, but the final ordering is a sort.

---

### 🔍 Scenario 3: Database ORDER BY — How It Works Internally

**Interviewer:** _"When you write `SELECT * FROM users ORDER BY age`, what happens internally?"_

**Answer:**
- If an **index exists on `age`**: the B-Tree index is already sorted → just traverse in order. O(log n + k).
- If **no index**: database performs a **file sort**:
  - If data fits in memory → **Quick Sort** (MySQL, PostgreSQL use variants).
  - If data too large → **External Merge Sort** (sort chunks on disk, k-way merge).
- `ORDER BY col1, col2` requires **stable sort** or a composite comparator.

| Query | Index? | Sort Method | Time |
|-------|--------|-------------|------|
| `ORDER BY id` | Primary key index | No sort needed | O(k) |
| `ORDER BY age` | Index on age | No sort needed | O(k) |
| `ORDER BY age` | No index | Quick Sort / External Sort | O(n log n) |
| `ORDER BY age, name` | Index on (age, name) | No sort needed | O(k) |

---

### 🔍 Scenario 4: E-Commerce — Multi-Criteria Product Sorting

**Interviewer:** _"On Amazon, users sort by 'Price: Low to High', then products with the same price are ordered by rating. How?"_

**Answer:**
- Use a **custom comparator** with tied-breaking:
```python
sort(products, key=lambda p: (p.price, -p.rating))
```
- Alternatively: **stable sort by rating first, then stable sort by price** — stability preserves the rating order within same-price groups.
- For server-side: database `ORDER BY price ASC, rating DESC`.

---

### 🔍 Scenario 5: Real-Time Leaderboard

**Interviewer:** _"Your game has a real-time leaderboard with millions of players. Scores update frequently. How do you maintain a sorted leaderboard?"_

**Answer:**
- **Don't re-sort on every update.** Use a data structure that maintains sorted order:
  - **Balanced BST (Red-Black Tree)** → O(log n) insert, delete, rank queries.
  - **Skip List** (used by Redis `ZSET`) → O(log n) sorted set with range queries.
- Redis `SORTED SET` (`ZADD`, `ZRANGE`, `ZRANK`) does exactly this internally with a skip list + hash map.
- **Top K leaderboard:** Only maintain a min-heap of size K → O(n log K).

---

### 🔍 Scenario 6: Log Analysis — Sorting Distributed Logs

**Interviewer:** _"You have log files from 100 servers. Each file is sorted by timestamp. How do you create one unified sorted log?"_

**Answer:**
- **K-way merge** using a **min-heap** of size K (K = 100 servers).
- Take the earliest timestamp from each file, push to heap.
- Extract min, output it, push the next entry from that file.
- Time: O(N log K), where N = total entries, K = 100 files. Much better than O(N log N).

```
File 1: [t=1.0, t=3.0, t=5.0, ...]
File 2: [t=1.5, t=2.0, t=4.0, ...]
File 3: [t=0.5, t=6.0, ...]

Min-Heap: [0.5(F3), 1.0(F1), 1.5(F2)]
Output 0.5 → push 6.0(F3)
Min-Heap: [1.0(F1), 1.5(F2), 6.0(F3)]
Output 1.0 → push 3.0(F1)
...
```

**Real-world:** This is exactly how Elasticsearch merges segment files and how Hadoop's MapReduce shuffle phase works.

---

### 🔍 Scenario 7: CI/CD — Dependency Resolution

**Interviewer:** _"Your build system has packages with dependencies. How do you determine the build order?"_

**Answer:**
- This is **Topological Sort** — a sorting algorithm for directed acyclic graphs (DAGs).
- Build a DAG: `package A depends on B` → edge B → A.
- Topological sort gives a valid build order.
- If a cycle is detected → error: circular dependency.
- Time: O(V + E).

```
Dependencies:
  C depends on A, B
  B depends on A
  D depends on C

DAG: A → B → C → D
    A → C

Topological Order: A, B, C, D ✅
```

**Used by:** npm, pip, Maven, Make, Webpack, Kubernetes (init containers).

---

### 🔍 Scenario 8: Version Sorting — Semantic Versioning

**Interviewer:** _"Your package manager needs to sort versions like '1.2.3', '1.10.1', '2.0.0'. String sort gives wrong order ('1.10' < '1.2' alphabetically). How?"_

**Answer:**
- Parse version into tuple of integers: `"1.10.1" → (1, 10, 1)`.
- Sort tuples lexicographically (compare major, then minor, then patch).
- Custom comparator:

```cpp
sort(versions.begin(), versions.end(), [](string& a, string& b) {
    auto parse = [](string& v) {
        vector<int> parts;
        stringstream ss(v); string token;
        while (getline(ss, token, '.')) parts.push_back(stoi(token));
        return parts;
    };
    return parse(a) < parse(b);
});
```

**Used by:** npm, Docker, GitHub Releases, package managers everywhere.

---

### 🔍 Scenario 9: Frontend — Sorting Table Columns

**Interviewer:** _"Your React table component allows users to sort by clicking column headers. How do you implement multi-column sorting?"_

**Answer:**
- Maintain sort state: `[{column: 'name', direction: 'asc'}, {column: 'age', direction: 'desc'}]`.
- Use **stable sort** with a chained comparator:

```javascript
data.sort((a, b) => {
    for (const {column, direction} of sortKeys) {
        const cmp = a[column] < b[column] ? -1 : a[column] > b[column] ? 1 : 0;
        if (cmp !== 0) return direction === 'asc' ? cmp : -cmp;
    }
    return 0;  // all keys equal
});
```

- **Performance:** For large tables (10K+ rows), sort in a **Web Worker** to avoid blocking the UI thread.
- **Virtual scrolling:** Only render visible rows after sorting.

---

### 🔍 Scenario 10: Map/Reduce — Distributed Sorting

**Interviewer:** _"How would you sort 1 TB of data across 100 machines?"_

**Answer:**
- **Sample + Partition:** Sample a subset to determine split points.
- **Distribute:** Send each record to the correct machine based on split points (range partitioning).
- **Local Sort:** Each machine sorts its partition in-memory (Quick Sort).
- **Concatenate:** Partitions are in order, so just concatenate.
- Total: O((N/M) × log(N/M)) per machine, where M = machines.

**This is how Google's MapReduce, Apache Spark, and Hadoop sort at scale.**

---

### 🔍 Scenario 11: Email — Priority Inbox

**Interviewer:** _"Gmail's Priority Inbox shows important emails first. How would you implement this?"_

**Answer:**
- Score each email: `importance = f(sender_reputation, subject_keywords, user_interaction_history)`.
- **Partial sort:** Only need top K important emails → min-heap of size K → O(n log K).
- **Online sorting:** As new emails arrive, insert into a **sorted structure** (balanced BST or heap) — no need to re-sort everything.
- Use **Insertion Sort** logic: each new email is inserted into its correct position in the sorted inbox.

---

### 🔍 Scenario 12: Data Pipeline — Stream Processing with Watermarks

**Interviewer:** _"Your streaming pipeline receives events out of order. How do you produce sorted output?"_

**Answer:**
- Buffer events in a **sorted window** (e.g., 5 seconds).
- Use **Insertion Sort** since most events arrive nearly in order (adaptive, O(n) on nearly sorted).
- After the watermark time passes, flush the buffer in sorted order.
- This is how **Apache Flink, Kafka Streams, and Google Cloud Dataflow** handle out-of-order events.
