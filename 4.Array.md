## 1. What is an Array?

An array is a **contiguous block of memory** that stores elements of the **same type**, accessible by **index** in O(1) time. It is the most fundamental data structure in computer science and the foundation for nearly every other data structure.

### Core Idea

> Store elements side-by-side in memory. Access any element instantly by computing its address: `base_address + index × element_size`.

```
Index:   0     1     2     3     4
Value: [ 10 | 20 | 30 | 40 | 50 ]
         ↑
    base address
```

### When to Use Arrays

- You need **O(1) random access** by index.
- You know the size in advance (or use dynamic arrays).
- You need **cache-friendly** sequential access.
- You need to implement other structures (stacks, queues, heaps, hash tables).

### Key Insight

> Arrays are **simple but powerful**. Most interview problems are array problems. Mastering array patterns is the highest-ROI skill for coding interviews.

## 3. Types of Arrays

| Type | Description | Size | Example (C++) |
|------|-------------|------|---------------|
| **Static Array** | Fixed size, allocated on stack | Compile-time | `int arr[5];` |
| **Dynamic Array** | Resizable, allocated on heap | Runtime | `vector<int> v;` |
| **2D Array** | Matrix / grid | m × n | `int grid[3][4];` |
| **Jagged Array** | Rows of different lengths | Variable | `vector<vector<int>>` |

### Static vs Dynamic Arrays

| Feature | Static (`int arr[]`) | Dynamic (`vector<int>`) |
|---------|---------------------|------------------------|
| Size | Fixed at compile time | Grows/shrinks at runtime |
| Memory | Stack (fast) | Heap (flexible) |
| Resize | Not possible | Automatic (amortized O(1) push_back) |
| Bounds checking | No | `.at()` throws exception |
| Use case | Known size, performance-critical | Unknown/variable size |

### How Dynamic Arrays (vector) Grow

1. Start with capacity `c` (e.g., 1 or 4).
2. When full, allocate a new array of size `2c`.
3. Copy all elements to the new array. Release old memory.
4. **Amortized O(1)** per insertion — most insertions are O(1), occasional O(n) copy.

```
push_back sequence:
  [1]           capacity=1
  [1,2]         capacity=2 (doubled, copied 1 element)
  [1,2,3]       capacity=4 (doubled, copied 2 elements)
  [1,2,3,4]     capacity=4
  [1,2,3,4,5]   capacity=8 (doubled, copied 4 elements)
```

## 4. Time & Space Complexity

| Operation | Array | vector (dynamic) | Notes |
|-----------|-------|-------------------|-------|
| Access by index | O(1) | O(1) | Fastest — this is the main advantage |
| Search (unsorted) | O(n) | O(n) | Linear scan |
| Search (sorted) | O(log n) | O(log n) | Binary search |
| Insert at end | N/A | O(1) amortized | `push_back` |
| Insert at beginning | O(n) | O(n) | Shift all elements |
| Insert at middle | O(n) | O(n) | Shift elements right |
| Delete at end | N/A | O(1) | `pop_back` |
| Delete at beginning | O(n) | O(n) | Shift all elements |
| Delete at middle | O(n) | O(n) | Shift elements left |
| Space | O(n) | O(n) | Contiguous memory |

### Why Arrays are Cache-Friendly

- Elements are stored **contiguously** in memory.
- CPU cache loads entire **cache lines** (64 bytes typically).
- Sequential access hits the cache → very fast.
- Linked lists have pointers scattered in memory → cache misses → slow.

## 5. C++ STL — vector & array

### vector (Dynamic Array) — Most Used

```cpp
#include <vector>

// Creation
vector<int> v;                    // empty
vector<int> v(5);                 // [0,0,0,0,0]
vector<int> v(5, 10);            // [10,10,10,10,10]
vector<int> v = {1, 2, 3, 4, 5}; // initializer list
vector<vector<int>> grid(m, vector<int>(n, 0)); // m×n matrix of 0s

// Add / Remove
v.push_back(10);     // add at end — O(1) amortized
v.pop_back();        // remove from end — O(1)
v.insert(v.begin() + i, val); // insert at index i — O(n)
v.erase(v.begin() + i);       // remove at index i — O(n)

// Access
v[0];       // no bounds check — O(1)
v.at(0);    // with bounds check — O(1)
v.front();  // first element
v.back();   // last element

// Size
v.size();     // current number of elements
v.empty();    // true if size == 0
v.capacity(); // allocated capacity (≥ size)
v.resize(10); // resize to 10 (fills with 0)
v.reserve(100); // pre-allocate capacity (avoids rehashing)

// Iterate
for (int i = 0; i < v.size(); i++) cout << v[i];  // index-based
for (int x : v) cout << x;                         // range-based
for (auto it = v.begin(); it != v.end(); it++) cout << *it; // iterator

// Sort
sort(v.begin(), v.end());            // ascending
sort(v.begin(), v.end(), greater<>()); // descending
sort(v.begin(), v.end(), [](int a, int b){ return a > b; }); // custom

// Search
auto it = find(v.begin(), v.end(), target); // linear O(n)
bool found = binary_search(v.begin(), v.end(), target); // O(log n), must be sorted
auto lb = lower_bound(v.begin(), v.end(), target); // first >= target

// Other useful operations
reverse(v.begin(), v.end());
int mx = *max_element(v.begin(), v.end());
int mn = *min_element(v.begin(), v.end());
int total = accumulate(v.begin(), v.end(), 0);
```

### array (Fixed Size, C++11)

```cpp
#include <array>
array<int, 5> a = {1, 2, 3, 4, 5};
a.size();  // 5 (compile-time constant)
a[0];      // access
a.fill(0); // fill all with 0
```

## 6. Key Techniques & Building Blocks

### 6.1 Prefix Sum

Pre-compute cumulative sums to answer **range sum queries in O(1)**.

```cpp
// Build prefix sum
vector<int> prefix(n + 1, 0);
for (int i = 0; i < n; i++)
    prefix[i + 1] = prefix[i] + arr[i];

// Sum of arr[l..r] (inclusive)
int rangeSum = prefix[r + 1] - prefix[l];
```

**Applications:** Range sum queries, subarray sum problems, 2D prefix sums for matrix queries.

### 6.2 Difference Array

Apply **range updates in O(1)**, then reconstruct the array.

```cpp
// Add val to all elements in range [l, r]
diff[l] += val;
diff[r + 1] -= val;

// Reconstruct: prefix sum of diff array
for (int i = 1; i < n; i++)
    diff[i] += diff[i - 1];
```

**Applications:** Range increment operations, meeting room overlaps, flight booking.

### 6.3 Two Pointers

Use two indices to scan the array, reducing O(n²) to O(n).

```cpp
// Two Sum on sorted array
int left = 0, right = n - 1;
while (left < right) {
    int sum = arr[left] + arr[right];
    if (sum == target) return {left, right};
    else if (sum < target) left++;
    else right--;
}
```

### 6.4 Sliding Window

Maintain a window of fixed or variable size, sliding across the array.

```cpp
// Max sum subarray of size k
int windowSum = 0, maxSum = 0;
for (int i = 0; i < n; i++) {
    windowSum += arr[i];
    if (i >= k) windowSum -= arr[i - k]; // slide
    if (i >= k - 1) maxSum = max(maxSum, windowSum);
}
```

### 6.5 Kadane's Algorithm

Find **maximum subarray sum** in O(n).

```cpp
int maxSubarraySum(vector<int>& arr) {
    int maxSum = arr[0], currentSum = arr[0];
    for (int i = 1; i < arr.size(); i++) {
        currentSum = max(arr[i], currentSum + arr[i]);
        maxSum = max(maxSum, currentSum);
    }
    return maxSum;
}
```

### 6.6 Dutch National Flag (3-Way Partition)

Partition array into three parts in a single pass O(n).

```cpp
// Sort array of 0s, 1s, 2s
void sortColors(vector<int>& arr) {
    int low = 0, mid = 0, high = arr.size() - 1;
    while (mid <= high) {
        if (arr[mid] == 0) swap(arr[low++], arr[mid++]);
        else if (arr[mid] == 1) mid++;
        else swap(arr[mid], arr[high--]);
    }
}
```

## 7. Pattern Recognition (15 Patterns)

### Pattern 1: Two Pointers (Opposite Ends)

**What:** Two pointers starting from both ends, moving inward.  
**When:** Sorted array, pair finding, container/trapping problems.

**Problems:** Two Sum II, Container With Most Water, Trapping Rain Water, 3Sum, Valid Palindrome.

---

### Pattern 2: Two Pointers (Same Direction — Fast & Slow)

**What:** Two pointers moving in the same direction at different speeds.  
**When:** Remove duplicates in-place, partition, linked list cycle.

```cpp
// Remove duplicates from sorted array
int removeDuplicates(vector<int>& arr) {
    if (arr.empty()) return 0;
    int slow = 0;
    for (int fast = 1; fast < arr.size(); fast++) {
        if (arr[fast] != arr[slow]) {
            arr[++slow] = arr[fast];
        }
    }
    return slow + 1;
}
```

**Problems:** Remove Duplicates, Remove Element, Move Zeroes, Sort Colors.

---

### Pattern 3: Fixed-Size Sliding Window

**What:** Window of size k slides across the array.  
**When:** Max/min/average of k consecutive elements.

**Problems:** Max Sum Subarray of Size K, Max of All Subarrays of Size K, Sliding Window Maximum.

---

### Pattern 4: Variable-Size Sliding Window

**What:** Window grows/shrinks based on a condition.  
**When:** Longest/shortest subarray with a property (sum, distinct elements, etc.).

```cpp
// Smallest subarray with sum >= target
int minSubArrayLen(int target, vector<int>& nums) {
    int left = 0, sum = 0, minLen = INT_MAX;
    for (int right = 0; right < nums.size(); right++) {
        sum += nums[right];
        while (sum >= target) {
            minLen = min(minLen, right - left + 1);
            sum -= nums[left++];
        }
    }
    return minLen == INT_MAX ? 0 : minLen;
}
```

**Problems:** Minimum Size Subarray Sum, Longest Substring Without Repeating Characters, Fruit Into Baskets.

---

### Pattern 5: Prefix Sum

**What:** Pre-compute cumulative sums for O(1) range queries.  
**When:** Multiple range sum queries, subarray sum equals K.

**Problems:** Range Sum Query, Subarray Sum Equals K, Contiguous Array, Product of Array Except Self.

---

### Pattern 6: Kadane's Algorithm (Maximum Subarray)

**What:** Track current subarray sum; reset when it goes negative.  
**When:** Maximum/minimum subarray sum, largest rectangle.

**Problems:** Maximum Subarray, Maximum Product Subarray, Best Time to Buy and Sell Stock.

---

### Pattern 7: Sorting + Greedy

**What:** Sort the array first, then apply greedy logic.  
**When:** Interval problems, meeting rooms, assignment, merging.

**Problems:** Merge Intervals, Non-Overlapping Intervals, Meeting Rooms I & II, Assign Cookies.

---

### Pattern 8: Moore's Voting Algorithm (Majority Element)

**What:** Find element appearing > n/2 times in O(n) time, O(1) space.  
**Key:** Maintain a candidate and count. If count drops to 0, switch candidate.

```cpp
int majorityElement(vector<int>& nums) {
    int candidate = nums[0], count = 1;
    for (int i = 1; i < nums.size(); i++) {
        if (count == 0) { candidate = nums[i]; count = 1; }
        else if (nums[i] == candidate) count++;
        else count--;
    }
    return candidate; // verify if needed
}
```

**Problems:** Majority Element, Majority Element II (> n/3).

---

### Pattern 9: Cyclic Sort / Index as Hash

**What:** Place each element at its "correct" index (`arr[i] = i` or `arr[i] = i+1`).  
**When:** Finding missing/duplicate/first missing positive in range [1, n].

```cpp
// Find the missing number in [0, n]
int missingNumber(vector<int>& nums) {
    int n = nums.size();
    int i = 0;
    while (i < n) {
        if (nums[i] < n && nums[i] != i && nums[nums[i]] != nums[i])
            swap(nums[i], nums[nums[i]]);
        else i++;
    }
    for (int i = 0; i < n; i++)
        if (nums[i] != i) return i;
    return n;
}
```

**Problems:** Missing Number, Find All Duplicates, First Missing Positive, Find the Duplicate Number.

---

### Pattern 10: Next Greater / Smaller Element (Monotonic Stack)

**What:** For each element, find the next element that is greater (or smaller).  
**Key:** Use a stack that maintains monotonic order.

```cpp
vector<int> nextGreaterElement(vector<int>& arr) {
    int n = arr.size();
    vector<int> result(n, -1);
    stack<int> st; // stores indices
    for (int i = 0; i < n; i++) {
        while (!st.empty() && arr[st.top()] < arr[i]) {
            result[st.top()] = arr[i];
            st.pop();
        }
        st.push(i);
    }
    return result;
}
```

**Problems:** Next Greater Element I & II, Daily Temperatures, Largest Rectangle in Histogram, Trapping Rain Water.

---

### Pattern 11: Merge Intervals

**What:** Merge overlapping intervals after sorting by start time.  
**Key:** Sort, then compare current end with next start.

```cpp
vector<vector<int>> merge(vector<vector<int>>& intervals) {
    sort(intervals.begin(), intervals.end());
    vector<vector<int>> merged;
    for (auto& interval : intervals) {
        if (merged.empty() || merged.back()[1] < interval[0])
            merged.push_back(interval);
        else
            merged.back()[1] = max(merged.back()[1], interval[1]);
    }
    return merged;
}
```

**Problems:** Merge Intervals, Insert Interval, Non-Overlapping Intervals, Meeting Rooms.

---

### Pattern 12: Matrix Traversal Patterns

**What:** Traverse a 2D array in specific orders.

- **Spiral Order:** Layer-by-layer, peeling the matrix from outside.
- **Diagonal:** `i + j = constant` for each diagonal.
- **Zigzag:** Alternate direction on each diagonal.
- **Rotate 90°:** Transpose + reverse each row.

**Problems:** Spiral Matrix, Rotate Image, Diagonal Traverse, Set Matrix Zeroes.

---

### Pattern 13: In-Place Array Manipulation

**What:** Modify the array without extra space.  
**When:** The problem says "do it in-place" or O(1) extra space.

**Techniques:**
- Swap elements to their correct positions.
- Use negative marking: `arr[abs(arr[i]) - 1] *= -1` to mark "visited".
- Use the array itself as a hash map.

**Problems:** Rotate Array, Move Zeroes, Remove Element, Find All Duplicates in Array.

---

### Pattern 14: Subarray Problems (Sum / Product / Count)

**What:** Find subarrays satisfying a condition.

**Approaches by constraint:**
| Constraint | Approach | Time |
|-----------|----------|------|
| All positive, sum = K | Sliding window | O(n) |
| Any integers, sum = K | Prefix sum + hash map | O(n) |
| Count subarrays with property | AtMost(K) - AtMost(K-1) trick | O(n) |
| Max product | Modified Kadane's (track min too) | O(n) |

---

### Pattern 15: Stock Buy / Sell Problems

**What:** Maximize profit from buying and selling stocks.

| Variant | Approach | Time |
|---------|----------|------|
| 1 transaction | Track min price, max profit | O(n) |
| Unlimited transactions | Add all positive differences | O(n) |
| At most 2 transactions | DP with states | O(n) |
| At most K transactions | DP - O(nK) | O(nK) |
| With cooldown | DP with states | O(n) |
| With transaction fee | DP with states | O(n) |

## 9. Common Mistakes & Edge Cases

### ❌ Common Mistakes

| Mistake | Why it's wrong | Fix |
|---------|---------------|-----|
| Off-by-one in loops | Accessing `arr[n]` (out of bounds) | Use `i < n`, not `i <= n` |
| Not handling **empty array** | Crash on `arr[0]` | Check `if (arr.empty())` first |
| Integer overflow in sum | `arr[i] + arr[j]` overflows int | Use `long long` |
| Modifying array while iterating | Unexpected behavior | Use index-based iteration or copy |
| Forgetting to sort before two pointers | Two pointer technique requires sorted input | Sort first |
| Using `==` to compare arrays | Compares pointers in C, not content | Use `equal()` or loop |
| Assuming 0-indexed everywhere | Some problems use 1-based | Read problem carefully |

### 🧪 Edge Cases to Always Test

- **Empty array** `[]`
- **Single element** `[5]`
- **Two elements** `[1, 2]` or `[2, 1]`
- **All same elements** `[3, 3, 3, 3]`
- **Already sorted** (ascending and descending)
- **Negative numbers** and **zeros**
- **Very large values** (INT_MAX, overflow risk)
- **Array of size n with values in [1, n]** (cyclic sort problems)
- **Odd vs even length** (median, partition)

## 10. Interview Tips & Frequently Asked Concepts

### 💡 Top Interview Tips

1. **Arrays are the #1 topic in interviews.** ~40% of all coding questions involve arrays.

2. **Always consider these approaches first:**
   - Can I sort it? → O(n log n) + simpler logic.
   - Can I use a hash map? → O(n) with extra space.
   - Can I use two pointers? → O(n) for sorted arrays.
   - Can I use sliding window? → O(n) for subarray problems.
   - Can I use prefix sum? → O(1) range queries after O(n) preprocessing.

3. **Time-space trade-off:** "I can solve this in O(n²) time and O(1) space, or O(n) time with O(n) space using a hash map. Which do you prefer?"

4. **In-place vs extra space:** Always ask if in-place is required. It often simplifies or complicates the solution significantly.

5. **Start with brute force, then optimize.** Interviewers want to see your thought process.

### 🎯 Frequently Asked Interview Concepts

#### Q1: Array vs Linked List — when to use which?

**A:**
| Feature | Array | Linked List |
|---------|-------|-------------|
| Access by index | O(1) ✅ | O(n) |
| Insert/Delete at beginning | O(n) | O(1) ✅ |
| Insert/Delete at end | O(1) amortized (vector) | O(1) with tail pointer |
| Memory | Contiguous, cache-friendly ✅ | Scattered, poor cache |
| Size flexibility | Dynamic arrays resize | Always flexible |

**Use Array when:** Random access needed, cache performance matters, size is predictable.  
**Use Linked List when:** Frequent insertions/deletions at arbitrary positions, no random access needed.

#### Q2: How does a dynamic array (vector) work?

**A:** Starts with a capacity. When full, allocates 2x the space, copies elements, frees old memory. Amortized O(1) per push_back. The doubling strategy ensures total copy cost across n insertions is O(n), giving O(1) amortized.

#### Q3: What is the sliding window technique?

**A:** Maintain a "window" (subarray) defined by two pointers. Instead of recalculating from scratch for each position, update the result by adding the new element entering the window and removing the element leaving. Reduces O(n×k) to O(n).

#### Q4: Explain Kadane's Algorithm.

**A:** Tracks the maximum subarray sum ending at each position. At each step: either extend the current subarray or start fresh. `currentSum = max(arr[i], currentSum + arr[i])`. If `currentSum > maxSum`, update. One pass, O(n) time, O(1) space.

#### Q5: What is the prefix sum trick?

**A:** Build a prefix array where `prefix[i] = sum(arr[0..i-1])`. Then `sum(arr[l..r]) = prefix[r+1] - prefix[l]` in O(1). Essential for subarray sum problems. Combined with hash map, solves "count subarrays with sum = K" in O(n).

#### Q6: What is Moore's Voting Algorithm?

**A:** Finds the majority element (appearing > n/2 times) in O(n) time, O(1) space. Maintain a candidate and count. For each element: if it matches candidate, increment; else decrement. If count hits 0, switch candidate. The true majority element will survive.

#### Q7: How to rotate an array by K positions in O(1) space?

**A:** Three-step reverse:
1. Reverse the entire array.
2. Reverse first K elements.
3. Reverse remaining n-K elements.
Time: O(n), Space: O(1).

#### Q8: How to find duplicates in O(1) extra space?

**A:** For arrays with values in [1, n]:
- **Negative marking:** Visit index `abs(val)-1`, negate it. If already negative → duplicate.
- **Cyclic sort:** Place each value at index `val-1`. After sorting, misplaced values are duplicates.

#### Q9: Explain the Dutch National Flag algorithm.

**A:** Partitions array into three sections using three pointers (low, mid, high). Single pass, O(n) time, O(1) space. Used for Sort Colors (0s, 1s, 2s) and three-way partitioning in quicksort.

#### Q10: What's the difference between subarray, subsequence, and subset?

**A:**
| Term | Contiguous? | Order Maintained? | Example from [1,2,3,4] |
|------|------------|-------------------|------------------------|
| **Subarray** | ✅ Yes | ✅ Yes | [2,3,4] |
| **Subsequence** | ❌ No | ✅ Yes | [1,3,4] |
| **Subset** | ❌ No | ❌ No | {1,4,2} |

Count: Subarrays = n(n+1)/2, Subsequences = 2ⁿ, Subsets = 2ⁿ.

```
│  │   └── PREFIX SUM (or Segment Tree for updates)       │
│  ├── Sorting-related?                                    │
│  │   ├── 3 categories → DUTCH NATIONAL FLAG             │
│  │   ├── >n/2 frequency → MOORE'S VOTING               │
│  │   └── Intervals → SORT + MERGE                      │
│  ├── In-place modification?                              │
│  │   ├── Values in [1,n] → CYCLIC SORT / INDEX HASH    │
│  │   └── Reverse tricks → THREE-REVERSE TECHNIQUE       │
│  ├── Next greater/smaller?                               │
│  │   └── MONOTONIC STACK                                │
│  └── Matrix problem?                                     │
│      ├── Sorted → BINARY SEARCH / STAIRCASE             │
│      └── Traversal → SPIRAL / DIAGONAL patterns         │
│                                                          │
│  ALWAYS CONSIDER:                                        │
│  • Can I sort first?         → O(n log n) + simpler     │
│  • Can I use a hash map?     → O(n) time, O(n) space    │
│  • Can I use two pointers?   → O(n), need sorted        │
│  • Can I do it in one pass?  → Track running state      │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

---

### 🔑 Key Takeaways

1. **Arrays are the foundation** — most interview problems are array problems in disguise.
2. **Two Pointers** and **Sliding Window** are the most frequently used techniques.
3. **Prefix Sum + Hash Map** solves the hardest subarray problems in O(n).
4. **Kadane's Algorithm** is a must-know for max subarray problems.
5. **In-place techniques** (negative marking, cyclic sort, reverse trick) save space.
6. **Monotonic Stack** is essential for next greater/smaller element problems.
7. Always start brute force, then optimize — show your thought process.

## 11. Arrays in Software Development & Web Development — Real-World Scenarios

> In SDE/web-dev interviews, the interviewer describes a **real product situation** and asks: _"How would you implement this? What data structure? What's the complexity?"_ This section covers real-world scenarios where arrays and array techniques are the answer.

---

### 🔍 Scenario 1: Image Processing — Pixel Manipulation

**Interviewer:** _"Your web app has an image editor. The user applies a blur filter. How do you implement it? What data structure is the image stored in?"_

**Answer:**
- An image is a **2D array of pixels** (each pixel = RGB values).
- Blur = **sliding window** on the 2D matrix. For each pixel, average its neighbors in a k×k window.
- Use a **prefix sum** on the 2D array for efficient box blur: O(1) per pixel vs O(k²).

```
Image (3-channel 2D array):
  pixel[y][x] = {R: 128, G: 200, B: 50}

Box Blur (3x3 window):
  new_pixel[y][x] = average of pixel[y-1..y+1][x-1..x+1]

With 2D prefix sum:
  sum of any rectangle = O(1) → total blur = O(m × n)
  Without prefix sum: O(m × n × k²) — much slower for large k
```

**Real-world:** Canvas API, WebGL shaders, and Photoshop all process images as 2D arrays.

---

### 🔍 Scenario 2: Spreadsheet — Dynamic Grid with Formulas

**Interviewer:** _"You're building a spreadsheet app (like Google Sheets). How do you store the grid? How do you handle SUM(A1:A100)?"_

**Answer:**
- The grid is a **2D dynamic array** (`vector<vector<Cell>>`).
- For `SUM(A1:A100)`: use a **prefix sum** on column A → O(1) per query.
- When a cell changes: update the prefix sum array incrementally (difference update) → O(1).
- For sparse spreadsheets: use a **hash map** `{(row, col) → value}` instead of a full 2D array to save memory.

---

### 🔍 Scenario 3: Undo/Redo — Action History Stack

**Interviewer:** _"Your text editor or design tool needs undo/redo. How do you implement it?"_

**Answer:**
- Use an **array (or vector) as a stack** of actions.
- Maintain a pointer `current` to the latest action.
- **Undo:** decrement pointer, reverse the action at `actions[current]`.
- **Redo:** increment pointer, re-apply the action.
- All operations: O(1).

```
actions = ["type 'H'", "type 'i'", "delete", "type '!'"]
pointer = 3 (at "type '!'")

Undo → pointer = 2, reverse "type '!'"
Undo → pointer = 1, reverse "delete"
Redo → pointer = 2, re-apply "delete"
```

**Real-world:** VS Code, Figma, Photoshop — all use array-based command stacks.

---

### 🔍 Scenario 4: Event Timeline — Calendar / Scheduling

**Interviewer:** _"Users book meetings on a shared calendar. How do you detect overlapping meetings? How do you find free slots?"_

**Answer:**
- Store meetings as an **array of intervals** `[start, end]`.
- **Detect overlaps:** Sort by start time, check if `meetings[i].start < meetings[i-1].end` → O(n log n).
- **Merge overlapping:** Merge Intervals algorithm → O(n log n).
- **Find free slots:** Gaps between merged intervals.
- **Max concurrent meetings:** Difference array / sweep line → O(n log n).

```
Meetings: [[9,10], [9:30,11], [11,12]]
Sorted:   [[9,10], [9:30,11], [11,12]]
Overlap:  9:30 < 10 → YES, meetings 1 and 2 overlap
Merged:   [[9,11], [11,12]]
Free:     No gap (11 == 11), all day is booked 9-12
```

### 🔍 Scenario 5: Real-Time Dashboard — Moving Average

**Interviewer:** _"Your dashboard shows a real-time moving average of the last 100 data points. New data arrives every second. How do you compute this efficiently?"_

**Answer:**
- Use a **circular buffer** (fixed-size array with wrap-around).
- Maintain a running `sum`. On new data: subtract the oldest, add the newest.
- Average = `sum / min(count, windowSize)`.
- All operations: O(1) per update.

```cpp
class MovingAverage {
    vector<double> buffer;
    int head = 0, count = 0;
    double sum = 0;
    int windowSize;
public:
    MovingAverage(int size) : buffer(size, 0), windowSize(size) {}
    
    double next(double val) {
        sum -= buffer[head];
        buffer[head] = val;
        sum += val;
        head = (head + 1) % windowSize;
        count = min(count + 1, windowSize);
        return sum / count;
    }
};
```

**Real-world:** Stock tickers, IoT sensor dashboards, network monitoring.

### 🔍 Scenario 6: E-Commerce — Product Ranking & Pagination

**Interviewer:** _"Your e-commerce site shows products sorted by relevance. Users can filter by price, rating, or category. How do you handle sorting and pagination?"_

**Answer:**
- Store products in **sorted arrays** (one per sort criteria, or sort on demand).
- **Pagination:** Array slice `products[pageNum * pageSize ... (pageNum+1) * pageSize - 1]` → O(1) per page with O(n log n) sort.
- **Multi-criteria sort:** Custom comparator `sort(products, [](a, b) { return a.relevance > b.relevance || (a.relevance == b.relevance && a.price < b.price); })`.
- **Top K products:** Use partial sort or quickselect → O(n) average instead of O(n log n).

### 🔍 Scenario 7: Audio / Signal Processing — Streaming Data Buffers

**Interviewer:** _"Your music streaming app processes audio in real-time. How do you buffer and process audio frames?"_

**Answer:**
- Audio samples are stored in **arrays (buffers)** of fixed size (e.g., 1024 samples).
- **Ring buffer** for continuous streaming: producer writes, consumer reads.
- **FFT** operates on arrays of samples to compute frequency spectrum.
- **Sliding window** for effects like reverb, compression, equalization.

### 🔍 Scenario 8: Rate Monitoring — Spike Detection

**Interviewer:** _"Your monitoring system tracks request counts per second. Detect if any 10-second window has > 1000 requests (a traffic spike). How?"_

**Answer:**
- Store per-second counts in an **array** (circular buffer of 60 seconds).
- Use a **sliding window sum** of size 10 on this array.
- On each new second: add new count, subtract the count 10 seconds ago.
- If window sum > 1000 → alert.
- All operations: O(1) per second.

```
counts = [45, 50, 48, 200, 300, 250, 180, 50, 30, 20, ...]
window_sum(0..9) = 45+50+48+200+300+250+180+50+30+20 = 1173 > 1000 → ALERT!
```

### 🔍 Scenario 9: Game Development — Collision Detection Grid

**Interviewer:** _"In your 2D game, you have 10,000 objects. How do you efficiently detect which objects are colliding?"_

**Answer:**
- Divide the game world into a **2D grid array** (spatial partitioning).
- Each cell stores an **array of objects** in that region.
- Only check collisions between objects in the **same cell or adjacent cells**.
- Reduces from O(n²) brute force to O(n × k) where k = average objects per cell.

```
Grid (10x10):
  cell[3][5] = [player, enemy1, bullet3]
  cell[3][6] = [enemy2]
  
  Only check: player vs enemy1, player vs bullet3, etc.
  Skip: player vs enemy2 (different cell, not adjacent)
```

### 🔍 Scenario 10: Database — Columnar Storage

**Interviewer:** _"Why do analytical databases (like ClickHouse, BigQuery) store data in columnar format instead of row format? What's the underlying structure?"_

**Answer:**
- **Row storage:** Each row is an array `[name, age, salary, ...]`. Good for fetching entire records.
- **Columnar storage:** Each column is a separate array `ages = [25, 30, 28, ...]`. Good for aggregations.
- `SELECT AVG(salary)` on columnar: scan one contiguous array → **cache-friendly**, vectorized.
- On row storage: must skip over name, age for each row → many cache misses.

| Query Type | Row Store | Columnar Store |
|-----------|-----------|----------------|
| `SELECT * WHERE id=5` | O(1) ✅ | O(n columns) |
| `SELECT AVG(salary)` | O(n × row_size) | O(n) ✅ Cache-friendly |
| INSERT entire row | O(1) ✅ | O(n columns) |

### 🔍 Scenario 11: Frontend — Virtual DOM Diffing

**Interviewer:** _"React uses a Virtual DOM. How does it efficiently find what changed between the old and new component trees?"_

**Answer:**
- Component children are stored as **arrays**.
- React diffs old vs new arrays using a **two-pointer / keyed approach**.
- With `key` props: React builds a hash map of keys to quickly match old and new elements.
- Without keys: linear scan and comparison → O(n).
- The diffing is essentially an **array comparison / LCS-like** algorithm optimized with heuristics.

```
Old: [<A key="1"/>, <B key="2"/>, <C key="3"/>]
New: [<B key="2"/>, <A key="1"/>, <D key="4"/>]

With keys: B moved (reorder), A moved, C removed, D added
React: minimal DOM operations = 3 (move B, move A, add D, remove C)
```

### 🔍 Scenario 12: Log Rotation — Circular Buffer

**Interviewer:** _"Your server keeps the last 10,000 log entries in memory for quick debugging. Older logs are discarded. How do you implement this efficiently?"_

**Answer:**
- Use a **circular buffer** (fixed-size array with head pointer).
- New log overwrites the oldest entry.
- No memory allocation/deallocation — just overwrite in O(1).
- To dump all logs: start from `(head + 1) % size` and read `size` entries.

```
buffer[10000], head = 0

write(log):
    buffer[head] = log
    head = (head + 1) % 10000

// After 15,000 writes:
// buffer contains logs 5001..15000 (latest 10,000)
// head = 5000 (points to oldest entry)
```

**Real-world:** Linux kernel ring buffer (`dmesg`), Redis append-only buffer, network packet buffers.

### 🔍 Scenario 13: Task Queue — Priority Scheduling

**Interviewer:** _"Your web server processes tasks with different priorities. How do you ensure high-priority tasks run first?"_

**Answer:**
- Use a **heap** (which is a special array!). A binary heap is stored as a flat array.
- Parent of `i` = `(i-1)/2`, children = `2i+1`, `2i+2`.
- Insert: O(log n), Extract max: O(log n).
- The array representation makes heaps cache-friendly and efficient.

```
Heap array: [10, 8, 6, 4, 2, 5, 3]

As tree:       10
              /    \
            8        6
           / \      / \
          4   2    5   3

Parent of index 3 (value 4) = index (3-1)/2 = 1 (value 8) ✅
```

### 🔍 Scenario 14: Feature Flags — User Segmentation with Bit Arrays

**Interviewer:** _"You have 64 feature flags. Each user has some flags enabled. How do you efficiently store and check flag status?"_

**Answer:**
- Use a **bit array** (or bitmask) — a single `long long` stores 64 flags.
- Check flag: `(flags >> i) & 1` → O(1).
- Enable flag: `flags |= (1 << i)` → O(1).
- Disable flag: `flags &= ~(1 << i)` → O(1).
- Space: 8 bytes per user for 64 flags (vs 64 bytes for a bool array).

```
User flags: 0b...00101101 = flags 0, 2, 3, 5 are ON

Check flag 3: (flags >> 3) & 1 = 1 → ON ✅
Enable flag 4: flags | (1 << 4) → 0b...00111101
```

**Real-world:** Unix file permissions, CPU flags, Redis bitmaps, bloom filters.

---

### 🔍 Scenario 15: Batch API Processing — Chunked Arrays

**Interviewer:** _"Your API needs to process 1 million records. You can't load all into memory at once. How do you handle it?"_

**Answer:**
- Process in **chunks** (subarrays of fixed size).
- Read chunk into array → process → write results → read next chunk.
- This is the **streaming / batching** pattern.

```
Total records: 1,000,000
Chunk size: 10,000

for offset in range(0, 1_000_000, 10_000):
    chunk = fetchRecords(offset, 10_000)  // array of 10K
    processed = processChunk(chunk)
    writeResults(processed)
    // chunk memory is reused/freed
```

**Real-world:** Database pagination, ETL pipelines, file upload processing, map-reduce.

### ⚖️ Trade-Off Discussions (Interviewers Love These)

| Question | Array | Alternative | When to Prefer Array |
|----------|-------|-------------|---------------------|
| _"Why not a linked list?"_ | O(1) access, cache-friendly | O(1) insert/delete at head | When random access or cache matters |
| _"Why not a hash map?"_ | O(1) access, less overhead | O(1) access, flexible keys | When keys are sequential integers |
| _"Why not a tree?"_ | Simpler, cache-friendly, less memory | O(log n) insert + search | When data is static or append-only |
| _"Why not a database?"_ | In-memory, fastest access | Persistent, ACID, scalable | When data fits in memory and speed is critical |
| _"Why use a circular buffer?"_ | Fixed memory, O(1), no GC pressure | Dynamic array grows indefinitely | When only recent N items matter |
| _"Why columnar over row storage?"_ | Column-wise aggregation is faster | Row-wise CRUD is faster | When workload is analytics-heavy (OLAP) |
