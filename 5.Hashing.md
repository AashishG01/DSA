## 1. What is Hashing?

Hashing is a technique that maps data of **arbitrary size** to data of **fixed size** using a **hash function**. It enables **near-constant time** O(1) lookups, insertions, and deletions — making it one of the most widely used data structures in software engineering.

### Core Idea

> Given a key, compute a **hash value** (integer), use it as an **index** in an array (called a hash table), and store/retrieve the value at that index.

```
Key → Hash Function → Index → Store/Retrieve Value
"apple" → hash("apple") → 3 → table[3] = "apple"
```

### When to Use Hashing

- You need **O(1) average** lookup, insert, or delete.
- You need to check **existence** of an element quickly.
- You need to **count frequencies** of elements.
- You need to find **pairs, duplicates, or groups** efficiently.
- You need to **map one value to another** (key-value association).

### Key Insight

> Hashing trades **space** for **speed**. You use extra memory (the hash table) to achieve near-constant time operations that would otherwise require O(n) linear scan or O(log n) binary search.

## 3. How Hashing Works

### Step-by-Step

1. **Hash Function** computes an integer hash code from the key.
2. **Compression** maps the hash code to a valid index: `index = hashCode % tableSize`.
3. **Store** the key-value pair at `table[index]`.
4. **Collision** occurs when two different keys map to the same index → resolve using chaining or open addressing.
5. **Retrieve** by computing the same hash, going to the index, and finding the key.

### Visual Walkthrough

```
Keys: "apple", "banana", "cherry", "date"
Table Size: 7

hash("apple")  % 7 = 3  → table[3] = "apple"
hash("banana") % 7 = 5  → table[5] = "banana"
hash("cherry") % 7 = 3  → COLLISION! Chain at table[3]: "apple" → "cherry"
hash("date")   % 7 = 1  → table[1] = "date"

Table:
[0] → null
[1] → "date"
[2] → null
[3] → "apple" → "cherry"  (chained)
[4] → null
[5] → "banana"
[6] → null
```

## 4. Hash Functions

### What Makes a Good Hash Function?

| Property | Description |
|----------|-------------|
| **Deterministic** | Same input always produces same output |
| **Uniform Distribution** | Spreads keys evenly across the table |
| **Fast to Compute** | O(1) or O(key length) |
| **Minimizes Collisions** | Different keys rarely map to the same index |
| **Avalanche Effect** | Small change in input → large change in output |

### Common Hash Functions

#### For Integers

```cpp
// Simple modulo
int hash(int key, int tableSize) {
    return key % tableSize;  // tableSize should be prime
}

// Multiplicative hashing (Knuth's method)
int hash(int key, int tableSize) {
    double A = 0.6180339887; // (sqrt(5) - 1) / 2
    return (int)(tableSize * fmod(key * A, 1.0));
}
```

#### For Strings

```cpp
// Polynomial rolling hash (most common in interviews)
int hash(string s, int tableSize) {
    long long hashVal = 0;
    int p = 31;       // prime base
    long long pPow = 1;
    for (char c : s) {
        hashVal = (hashVal + (c - 'a' + 1) * pPow) % tableSize;
        pPow = (pPow * p) % tableSize;
    }
    return hashVal;
}
```

#### Why Use Prime Numbers for Table Size?

- Primes distribute hash values more uniformly.
- Non-prime sizes cause patterns in modulo results, increasing collisions.
- Example: tableSize = 10 only uses last digit; tableSize = 11 (prime) uses all digits.

## 5. Collision Handling

### What is a Collision?

A collision occurs when **two different keys** produce the **same index**: `hash(key1) % size == hash(key2) % size`.

### Method 1: Separate Chaining (Open Hashing)

Each bucket is a **linked list** (or dynamic array). Colliding elements are appended to the chain.

```
Index 3: "apple" → "cherry" → "grape"  (3 elements chained)
```

| Pros | Cons |
|------|------|
| Simple to implement | Extra memory for pointers |
| Works well with high load factors | Poor cache performance (pointer chasing) |
| Deletion is straightforward | Worst case O(n) if all collide |

### Method 2: Open Addressing (Closed Hashing)

All elements stored **inside the table**. On collision, probe for the next available slot.

#### Linear Probing

```
index = (hash(key) + i) % tableSize    // i = 0, 1, 2, 3, ...
```

Problem: **Primary clustering** — long runs of occupied slots form.

#### Quadratic Probing

```
index = (hash(key) + i²) % tableSize   // i = 0, 1, 2, 3, ...
```

Problem: **Secondary clustering** — keys with same initial hash follow same probe sequence.

#### Double Hashing

```
index = (hash1(key) + i * hash2(key)) % tableSize
```

Best distribution among open addressing methods. `hash2(key)` should never return 0.

### Comparison Table

| Feature | Separate Chaining | Open Addressing |
|---------|-------------------|-----------------|
| Load factor > 1 | ✅ Allowed | ❌ Must be < 1 |
| Cache performance | Poor (pointers) | Good (contiguous) |
| Deletion | Easy | Tricky (need tombstones) |
| Implementation | Simpler | More complex |
| Used in | Java HashMap | Python dict, C++ unordered_map (varies) |

### Load Factor

```
Load Factor (α) = number of elements / table size
```

- **Chaining:** α can exceed 1, but performance degrades.
- **Open Addressing:** α must stay below ~0.7-0.8; rehashing triggers when exceeded.
- **Rehashing:** When load factor exceeds threshold, create a new table (usually 2x size), reinsert all elements. Amortized O(1).

## 6. Time & Space Complexity

| Operation | Average Case | Worst Case | Notes |
|-----------|-------------|------------|-------|
| Insert | O(1) | O(n) | Worst case: all keys collide |
| Search | O(1) | O(n) | Worst case: all in same bucket |
| Delete | O(1) | O(n) | Same as search |
| Space | O(n) | O(n) | n = number of stored elements |

### Why O(1) Average?

With a good hash function and reasonable load factor (α < 0.75):
- Expected chain length (chaining) = α ≈ constant.
- Expected probe length (open addressing) = 1 / (1 - α) ≈ constant.

### When Does Worst Case O(n) Happen?

- **Bad hash function** that maps everything to the same bucket.
- **Adversarial input** designed to cause collisions (competitive programming, security attacks).
- **Very high load factor** without rehashing.

### Amortized Analysis of Rehashing

- Rehashing happens when load factor exceeds threshold.
- Doubles the table size and reinserts all n elements: O(n).
- But this only happens every ~n insertions.
- **Amortized cost per insertion = O(1).**

## 7. Hash Map vs Hash Set

| Feature | Hash Map | Hash Set |
|---------|----------|----------|
| Stores | Key-Value pairs | Only Keys |
| Use case | Map/associate values | Check existence / uniqueness |
| Example | Word → Frequency | Unique elements |
| C++ STL | `unordered_map` | `unordered_set` |
| Java | `HashMap` | `HashSet` |
| Python | `dict` | `set` |

### When to Use Which?

- **Hash Set:** "Does this element exist?" / "Find all unique elements."
- **Hash Map:** "How many times does this appear?" / "What value is associated with this key?"

## 8. C++ STL — unordered_map & unordered_set

### unordered_map

```cpp
#include <unordered_map>

unordered_map<string, int> freq;

// Insert
freq["apple"] = 5;
freq.insert({"banana", 3});

// Access
cout << freq["apple"];       // 5
cout << freq.at("banana");   // 3

// Check existence
if (freq.find("cherry") != freq.end()) { /* found */ }
if (freq.count("apple")) { /* exists */ }

// Iterate
for (auto& [key, val] : freq) {
    cout << key << ": " << val << endl;
}

// Delete
freq.erase("apple");

// Size
cout << freq.size();
```

### unordered_set

```cpp
#include <unordered_set>

unordered_set<int> seen;

// Insert
seen.insert(10);
seen.insert(20);

// Check existence
if (seen.count(10)) { /* exists */ }
if (seen.find(20) != seen.end()) { /* found */ }

// Delete
seen.erase(10);

// Size
cout << seen.size();

// Iterate
for (int x : seen) {
    cout << x << " ";
}
```

### map vs unordered_map

| Feature | `map` (ordered) | `unordered_map` (hash) |
|---------|-----------------|------------------------|
| Underlying DS | Red-Black Tree | Hash Table |
| Insert/Search/Delete | O(log n) | O(1) average, O(n) worst |
| Ordered iteration | ✅ Yes | ❌ No |
| Memory | Less overhead | More (hash table + buckets) |
| Custom key type | Needs `operator<` | Needs custom hash function |
| Use when | Need sorted order | Need fastest access |

### Custom Hash for User-Defined Types

```cpp
struct Point {
    int x, y;
    bool operator==(const Point& other) const {
        return x == other.x && y == other.y;
    }
};

struct PointHash {
    size_t operator()(const Point& p) const {
        size_t h1 = hash<int>()(p.x);
        size_t h2 = hash<int>()(p.y);
        return h1 ^ (h2 << 1);  // combine hashes
    }
};

unordered_map<Point, string, PointHash> pointMap;
```

## 9. Pattern Recognition (12 Patterns)

### Pattern 1: Frequency Count

**What:** Count occurrences of each element.  
**Key:** `map[element]++`

```cpp
vector<int> arr = {1, 2, 3, 2, 1, 3, 3};
unordered_map<int, int> freq;
for (int x : arr) freq[x]++;
// freq = {1:2, 2:2, 3:3}
```

**Problems:** Most frequent element, frequency sort, top K frequent elements.

---

### Pattern 2: Existence Check / Duplicate Detection

**What:** Check if element was seen before.  
**Key:** Use a `set` to track seen elements.

```cpp
bool hasDuplicate(vector<int>& arr) {
    unordered_set<int> seen;
    for (int x : arr) {
        if (seen.count(x)) return true;
        seen.insert(x);
    }
    return false;
}
```

**Problems:** Contains Duplicate, First Repeating Element, Happy Number.

---

### Pattern 3: Two Sum / Pair Finding

**What:** Find two elements with a specific relationship (sum, difference, etc.).  
**Key:** For each element, check if its complement exists in the map.

```cpp
vector<int> twoSum(vector<int>& nums, int target) {
    unordered_map<int, int> seen; // value → index
    for (int i = 0; i < nums.size(); i++) {
        int complement = target - nums[i];
        if (seen.count(complement))
            return {seen[complement], i};
        seen[nums[i]] = i;
    }
    return {};
}
```

**Problems:** Two Sum, Four Sum, Pair with given difference, Count pairs with sum.

---

### Pattern 4: Subarray Sum / Prefix Sum + Hash Map

**What:** Find subarrays with a target sum using prefix sums and a hash map.  
**Key:** `prefixSum[j] - prefixSum[i] == target` → check if `prefixSum[j] - target` exists in the map.

```cpp
int subarraySum(vector<int>& nums, int k) {
    unordered_map<int, int> prefixCount;
    prefixCount[0] = 1;  // empty prefix
    int sum = 0, count = 0;
    for (int x : nums) {
        sum += x;
        if (prefixCount.count(sum - k))
            count += prefixCount[sum - k];
        prefixCount[sum]++;
    }
    return count;
}
```

**This is one of the MOST important patterns in interviews.**

**Problems:** Subarray Sum Equals K, Contiguous Array (0s and 1s), Longest Subarray with Sum K, Count of Subarrays with XOR K.

---

### Pattern 5: Grouping / Anagram Grouping

**What:** Group elements that share a common property.  
**Key:** Compute a **canonical key** for each element, use it as the map key.

```cpp
vector<vector<string>> groupAnagrams(vector<string>& strs) {
    unordered_map<string, vector<string>> groups;
    for (string& s : strs) {
        string key = s;
        sort(key.begin(), key.end());  // canonical form
        groups[key].push_back(s);
    }
    vector<vector<string>> result;
    for (auto& [key, group] : groups)
        result.push_back(group);
    return result;
}
```

**Problems:** Group Anagrams, Group Shifted Strings, Find All Duplicates.

---

### Pattern 6: Sliding Window + Hash Map

**What:** Track character/element frequencies in a window.  
**Key:** Maintain a hash map of the current window; update on slide.

```cpp
// Longest substring without repeating characters
int lengthOfLongestSubstring(string s) {
    unordered_map<char, int> lastIndex;
    int maxLen = 0, start = 0;
    for (int i = 0; i < s.size(); i++) {
        if (lastIndex.count(s[i]) && lastIndex[s[i]] >= start)
            start = lastIndex[s[i]] + 1;
        lastIndex[s[i]] = i;
        maxLen = max(maxLen, i - start + 1);
    }
    return maxLen;
}
```

**Problems:** Longest Substring Without Repeating Characters, Minimum Window Substring, Permutation in String, Longest Substring with At Most K Distinct Characters.

---

### Pattern 7: String Matching — Rolling Hash (Rabin-Karp)

**What:** Find a pattern in a text using hashing for O(n) average time.  
**Key:** Compute hash of the pattern and a sliding hash of the text window. Compare hashes; verify on match.

```cpp
// Rabin-Karp Algorithm
bool rabinKarp(string text, string pattern) {
    int n = text.size(), m = pattern.size();
    if (m > n) return false;
    
    long long base = 31, mod = 1e9 + 7;
    long long patternHash = 0, windowHash = 0, power = 1;
    
    // Compute hash of pattern and first window
    for (int i = 0; i < m; i++) {
        patternHash = (patternHash + (pattern[i] - 'a' + 1) * power) % mod;
        windowHash = (windowHash + (text[i] - 'a' + 1) * power) % mod;
        if (i < m - 1) power = (power * base) % mod;
    }
    
    for (int i = 0; i <= n - m; i++) {
        if (patternHash == windowHash) {
            // Verify character by character (handle hash collision)
            if (text.substr(i, m) == pattern) return true;
        }
        // Slide window
        if (i < n - m) {
            windowHash = (windowHash - (text[i] - 'a' + 1)) / base;
            windowHash = (windowHash + (text[i + m] - 'a' + 1) * power) % mod;
        }
    }
    return false;
}
```

**Problems:** Implement strStr(), Repeated String Match, Longest Duplicate Substring.

---

### Pattern 8: Hash Map as Cache / Memoization

**What:** Cache computed results to avoid recalculation.  
**Key:** `if (cache.count(key)) return cache[key]; else compute and store`.

```cpp
unordered_map<int, int> memo;
int fib(int n) {
    if (n <= 1) return n;
    if (memo.count(n)) return memo[n];
    return memo[n] = fib(n - 1) + fib(n - 2);
}
```

**Problems:** Fibonacci, Climbing Stairs, Word Break, All DP problems with memoization.

### Pattern 9: Set Operations — Union, Intersection, Difference

**What:** Perform set operations on collections.  
**Key:** Insert into sets, then iterate and check.

```cpp
// Intersection of two arrays
vector<int> intersection(vector<int>& a, vector<int>& b) {
    unordered_set<int> setA(a.begin(), a.end());
    vector<int> result;
    for (int x : b)
        if (setA.count(x)) {
            result.push_back(x);
            setA.erase(x); // avoid duplicates
        }
    return result;
}
```

**Problems:** Intersection of Two Arrays I & II, Union of Two Arrays, Missing Number.

### Pattern 10: Graph Problems — Adjacency Representation with Hash Map

**What:** Represent graphs where nodes aren't simple integers.  
**Key:** `unordered_map<Node, vector<Node>>` for adjacency list.

```cpp
// Clone Graph — map original node to cloned node
unordered_map<Node*, Node*> visited;
Node* cloneGraph(Node* node) {
    if (!node) return nullptr;
    if (visited.count(node)) return visited[node];
    Node* clone = new Node(node->val);
    visited[node] = clone;
    for (Node* neighbor : node->neighbors)
        clone->neighbors.push_back(cloneGraph(neighbor));
    return clone;
}
```

**Problems:** Clone Graph, Course Schedule, Number of Islands (with coordinate hashing).

### Pattern 11: Linked List Problems — Cycle Detection with Hash Set

**What:** Detect cycles or find intersections in linked lists.  
**Key:** Store visited nodes in a hash set.

```cpp
bool hasCycle(ListNode* head) {
    unordered_set<ListNode*> visited;
    while (head) {
        if (visited.count(head)) return true;
        visited.insert(head);
        head = head->next;
    }
    return false;
}
```

**Note:** Floyd's algorithm (two pointers) does this in O(1) space, but hash set is the intuitive approach and often asked as the first solution.

**Problems:** Linked List Cycle, Intersection of Two Linked Lists, Find the Duplicate Number.

### Pattern 12: Design Problems — LRU Cache, HashMap from Scratch

**What:** Design data structures that use hashing internally.  
**Key:** Combine hash map with other structures (doubly linked list, arrays).

```cpp
// LRU Cache = Hash Map + Doubly Linked List
class LRUCache {
    int capacity;
    list<pair<int, int>> cache;  // DLL: {key, value}
    unordered_map<int, list<pair<int, int>>::iterator> map;  // key → DLL iterator
    
public:
    LRUCache(int cap) : capacity(cap) {}
    
    int get(int key) {
        if (!map.count(key)) return -1;
        cache.splice(cache.begin(), cache, map[key]);  // move to front
        return map[key]->second;
    }
    
    void put(int key, int value) {
        if (map.count(key)) {
            map[key]->second = value;
            cache.splice(cache.begin(), cache, map[key]);
        } else {
            if (cache.size() == capacity) {
                map.erase(cache.back().first);
                cache.pop_back();
            }
            cache.push_front({key, value});
            map[key] = cache.begin();
        }
    }
};
```

**Problems:** LRU Cache, LFU Cache, Design HashMap, Design HashSet, Insert Delete GetRandom O(1).

## 11. Common Mistakes & Edge Cases

### ❌ Common Mistakes

| Mistake | Why it's wrong | Fix |
|---------|---------------|-----|
| Using `map[]` to check existence | Creates a default entry if key doesn't exist | Use `map.find()` or `map.count()` |
| Forgetting that `unordered_map` is **not ordered** | Iterating gives random order | Use `map` if order matters |
| Not handling **hash collisions** in custom hash | Wrong results for equal keys | Implement both `hash` and `operator==` |
| Using `double` / `float` as hash key | Floating point precision issues | Multiply by 10^k and use int, or use string representation |
| Assuming O(1) always | Worst case is O(n) with bad hash | Use `unordered_map` with reserve, or switch to `map` for guaranteed O(log n) |
| Not considering **negative numbers** in modulo | `-5 % 3` may give `-2` in C++ | Use `((key % size) + size) % size` |
| Modifying a set/map while iterating | Undefined behavior / crash | Collect keys to modify first, then modify |

### 🧪 Edge Cases to Always Test

- **Empty input** — empty array, empty string
- **Single element** input
- **All elements the same** — frequency count edge case
- **Negative numbers** — affects hash computation
- **Very large values** — potential integer overflow in hash
- **String keys with special characters** — Unicode handling
- **Collision-heavy inputs** — worst case testing
- **Key not found** — handle gracefully
- **Duplicate keys** — understand overwrite vs accumulate behavior

## 12. Interview Tips & Frequently Asked Concepts

### 💡 Top Interview Tips

1. **Default to hash map** when you need O(1) lookup. It's the single most useful data structure in interviews.

2. **Prefix Sum + Hash Map** is the #1 pattern you MUST know. It appears in ~15% of medium interview questions.

3. **Think "complement"**: For pair problems, don't check all pairs O(n²). For each element, check if its complement exists in the map → O(n).

4. **State your trade-offs**: "I'm using extra O(n) space with a hash map to bring the time from O(n²) to O(n)."

5. **Know when NOT to use hashing**: If you need sorted data, range queries, or worst-case guarantees → use a balanced BST (`map`/`set` in C++).

6. **Counting pattern**: `map[x]++` on insert, `if (map[x] == 0) map.erase(x)` on remove — keeps the map clean.

### 🎯 Frequently Asked Interview Concepts

#### Q1: What is the time complexity of hash map operations?

**A:** **Average O(1)** for insert, search, delete. **Worst case O(n)** when all keys hash to the same bucket. With a good hash function and load factor < 0.75, amortized O(1) is guaranteed.

#### Q2: How does a hash map work internally?

**A:**
1. Apply hash function to the key → get an integer.
2. Modulo with table size → get bucket index.
3. If bucket is empty, store key-value pair.
4. If collision, resolve via chaining (linked list at that bucket) or open addressing (probe for next empty slot).
5. When load factor exceeds threshold (~0.75), rehash: create 2x table, reinsert all elements.

#### Q3: What's the difference between HashMap and HashSet?

**A:**
- **HashMap** stores key-value pairs, used for mapping/association.
- **HashSet** stores only keys, used for existence checking and uniqueness.
- Internally, HashSet is often a HashMap where the value is a dummy.

#### Q4: How do you handle collisions?

**A:** Two main approaches:
- **Separate Chaining**: Each bucket has a linked list. Colliding elements are appended. Simple, allows load factor > 1.
- **Open Addressing**: Colliding elements probe for next available slot (linear, quadratic, or double hashing). Better cache performance, but load factor must be < 1.

#### Q5: What is rehashing? When does it happen?

**A:** When load factor (elements/table_size) exceeds a threshold (typically 0.75):
1. Create a new table, usually 2x the size.
2. Recompute hash for every existing key using the new table size.
3. Insert all elements into the new table.
- **Cost:** O(n) for one rehash, but **amortized O(1)** per insertion since it happens infrequently.

#### Q6: unordered_map vs map — when to use which?

**A:**
- `unordered_map`: O(1) average, no ordering. Use when you need **fastest access**.
- `map`: O(log n) guaranteed, sorted by key. Use when you need **ordered iteration, lower_bound, or range queries**.
- In competitive programming: `unordered_map` can get TLE with adversarial test cases → sometimes safer to use `map`.

#### Q7: How would you design a Hash Map from scratch?

**A:** (Very popular interview question at FAANG)
1. Choose a prime table size.
2. Implement a hash function (polynomial hash for strings, modulo for ints).
3. Use an array of buckets (linked lists for chaining).
4. Implement `put(key, value)`, `get(key)`, `remove(key)`.
5. Track load factor; rehash when > 0.75.

#### Q8: What is the Prefix Sum + Hash Map trick?

**A:**
- Maintain running `prefix_sum`.
- Store `{prefix_sum: count}` in a hash map.
- To find subarrays with sum = k: check if `prefix_sum - k` exists in the map.
- Works because: if `prefix[j] - prefix[i] == k`, then there's a subarray `[i+1, j]` with sum k.

#### Q9: How do you hash custom objects?

**A:**
- Define a hash function that combines the hashes of all relevant fields.
- Common approach: XOR with bit shifting `h1 ^ (h2 << 1)`.
- Better: Use `boost::hash_combine` or polynomial combination.
- Also define `operator==` for equality comparison.

#### Q10: Hash Map vs Array for frequency counting — when to use which?

**A:**
- **Array**: When keys are small integers with known range (e.g., characters a-z → `freq[26]`). Faster, less memory.
- **Hash Map**: When keys are large, sparse, or non-integer (strings, negative numbers, IDs). More flexible.

1. Hashing gives **O(1) average** time for insert, search, delete — the fastest general-purpose data structure.

## 13. Hashing in Software Development & Web Development — Real-World Scenarios

> In SDE/web-dev interviews, the interviewer describes a **real product situation** and asks: _"Which data structure would you use? Why? What's the complexity?"_ This section covers real-world scenarios where hashing is the answer.

---

### 🔍 Scenario 1: User Authentication — Password Storage

**Interviewer:** _"Your website stores user passwords. A user logs in with their password. How do you verify it without storing the actual password?"_

**Answer:**
- **Never store plain-text passwords.** Store the **hash** of the password.
- On registration: `store(hash(password + salt))`.
- On login: `hash(inputPassword + salt) == storedHash?`
- Use cryptographic hash functions: **bcrypt, scrypt, or Argon2** (NOT MD5 or SHA-1).
- Time: O(1) for lookup, hash computation is O(password_length) but constant in practice.

```
Registration:
  password = "MyP@ss123"
  salt = random_bytes(16)
  stored_hash = bcrypt("MyP@ss123" + salt)
  DB: {username: "john", hash: "$2b$10$...", salt: "..."}

Login:
  input = "MyP@ss123"
  bcrypt(input + stored_salt) == stored_hash? → ✅ Authenticated
```

**Why hashing and not encryption?**
- **Hashing is one-way**: even if DB is breached, passwords can't be reversed.
- **Encryption is two-way**: if the key is stolen, all passwords are compromised.

**Follow-up topics:** Rainbow tables, salting, key stretching, bcrypt cost factor.

---

### 🔍 Scenario 2: Session Management — Token-Based Auth

**Interviewer:** _"After login, how does your server know the user is authenticated on subsequent requests without re-asking the password?"_

**Answer:**
- Generate a **session token** (random hash) and store it in a hash map: `{token → userInfo}`.
- Send the token to the client (cookie or header).
- On each request: look up the token in the hash map → O(1) to verify.

```
Server-side:
  sessions = HashMap<String, UserSession>()
  
  // On login:
  token = generateRandomHash()
  sessions[token] = {userId: 42, role: "admin", expiry: ...}
  response.setCookie("session", token)
  
  // On each request:
  token = request.getCookie("session")
  if sessions.containsKey(token):
      user = sessions[token]  // O(1) lookup
      // proceed with request
  else:
      return 401 Unauthorized
```

**Alternatives:** JWT (JSON Web Tokens) — stateless, no server-side hash map needed. Trade-off: can't revoke individual tokens easily.

### 🔍 Scenario 3: URL Shortener (like bit.ly)

**Interviewer:** _"Design a URL shortener. Given a long URL, generate a short code. Given a short code, redirect to the original URL."_

**Answer:**
- Use **two hash maps**:
  - `longToShort: HashMap<String, String>` — for deduplication.
  - `shortToLong: HashMap<String, String>` — for redirect lookups.
- Generate the short code by hashing the long URL (or using a counter + base62 encoding).
- Both lookups are O(1).

```
longToShort = {}
shortToLong = {}

// Shorten:
shortCode = base62Encode(hash(longURL))  // e.g., "abc123"
longToShort[longURL] = shortCode
shortToLong[shortCode] = longURL

// Redirect:
GET /abc123 → shortToLong["abc123"] → "https://example.com/very/long/path"
                                       → 302 Redirect
```

**System Design depth:** Discuss collision handling, distributed hash maps, read-heavy vs write-heavy, caching with hash maps.

### 🔍 Scenario 4: Web Caching — CDN & Browser Cache

**Interviewer:** _"How does a CDN decide if a requested resource is cached? How does the browser know if a file has changed?"_

**Answer:**
- **CDN/Server**: Maintains a hash map of `{URL → cached_response}`. On request: O(1) lookup. Cache miss → fetch from origin, store in hash map.
- **Browser**: Uses **ETags** — a hash of the file content.
  - Server sends `ETag: "abc123"` with the response.
  - Browser sends `If-None-Match: "abc123"` on next request.
  - Server compares hashes: if match → `304 Not Modified` (no data transfer).

```
First request:
  GET /style.css
  Response: ETag: "hash_of_file_v1", body: ...

Second request:
  GET /style.css
  If-None-Match: "hash_of_file_v1"
  Server: hash("style.css") == "hash_of_file_v1"? → 304 Not Modified (saves bandwidth)
```

**Real-world connection:** Every CDN (Cloudflare, Akamai, AWS CloudFront) uses hash maps for cache storage and ETags for cache validation.

### 🔍 Scenario 5: Database Indexing — Hash Indexes

**Interviewer:** _"When would you use a hash index vs a B-Tree index in a database?"_

**Answer:**

| Feature | Hash Index | B-Tree Index |
|---------|-----------|-------------|
| Exact lookup (`WHERE id = 5`) | O(1) ✅ Fastest | O(log n) |
| Range queries (`WHERE age > 25`) | ❌ Not supported | O(log n + k) ✅ |
| Ordered iteration | ❌ No | ✅ Yes |
| Prefix search (`LIKE 'app%'`) | ❌ No | ✅ Yes |

- **Use Hash Index**: When you ONLY need exact-match lookups (e.g., session tokens, user IDs).
- **Use B-Tree Index**: When you need range queries, sorting, or prefix matching.
- PostgreSQL supports both. MySQL InnoDB uses B-Tree by default; MEMORY engine supports hash.

### 🔍 Scenario 6: Rate Limiting — Token Bucket with Hash Map

**Interviewer:** _"Your API needs to rate-limit users to 100 requests per minute. You have millions of users. How do you track each user's request count?"_

**Answer:**
- Use a hash map: `{userId → {count, windowStart}}`.
- On each request: look up the user in O(1), check/update their counter.
- Fixed window: reset count when window expires.
- Sliding window log: store timestamps in a list per user (more memory but more accurate).

```
rateLimits = HashMap<String, {count: int, windowStart: timestamp}>()

function isAllowed(userId):
    if userId not in rateLimits:
        rateLimits[userId] = {count: 1, windowStart: now()}
        return true
    
    entry = rateLimits[userId]
    if now() - entry.windowStart > 60s:
        entry.count = 1
        entry.windowStart = now()
        return true
    
    if entry.count >= 100:
        return false  // Rate limited!
    
    entry.count++
    return true
```

**System Design depth:** Discuss distributed rate limiting with Redis hash maps, sliding window counters, token bucket algorithm.

### 🔍 Scenario 7: Distributed Systems — Consistent Hashing

**Interviewer:** _"You have a distributed cache with 10 servers. How do you decide which server stores a given key? What happens when a server goes down?"_

**Answer:**
- **Naive approach:** `server = hash(key) % numServers` → but adding/removing a server remaps almost ALL keys.
- **Consistent Hashing:** Place servers on a hash ring. Each key maps to the next server clockwise.
  - Adding a server: only keys between the new server and its predecessor are remapped.
  - Removing a server: only its keys move to the next server.
  - Impact: only `K/n` keys remapped (K = total keys, n = servers).

```
Hash Ring (0 to 360):
  Server A at hash 45
  Server B at hash 150
  Server C at hash 270

  key "user:123" → hash = 100 → next server clockwise = B
  key "user:456" → hash = 300 → next server clockwise = A (wraps around)

  If Server B dies:
    Only keys in range (45, 150] move to Server C
    Servers A and C are unaffected
```

**Used by:** Amazon DynamoDB, Apache Cassandra, Memcached, load balancers.

### 🔍 Scenario 8: Deduplication — Preventing Duplicate Submissions

**Interviewer:** _"A user double-clicks the 'Place Order' button. How do you ensure the order isn't placed twice?"_

**Answer:**
- Generate a unique **idempotency key** (UUID) on the client side.
- Server stores: `processedKeys = HashSet<String>()`.
- Before processing: check `if idempotencyKey in processedKeys → return cached result`.
- After processing: add key to the set.
- All operations: O(1) with hash set.

```
Client:
  idempotencyKey = uuid()  // Generated once per action
  POST /order  { key: "abc-123", items: [...] }
  POST /order  { key: "abc-123", items: [...] }  // Double click!

Server:
  processed = HashSet()
  
  if "abc-123" in processed:
      return cachedResponse["abc-123"]  // Don't process again
  
  // Process order...
  processed.add("abc-123")
  cachedResponse["abc-123"] = response
```

**Used by:** Stripe, PayPal, and almost every payment API uses idempotency keys backed by hash maps.

### 🔍 Scenario 9: Content-Addressable Storage — Git Internals

**Interviewer:** _"How does Git track file changes efficiently? How does it know if two files are identical?"_

**Answer:**
- Git stores every file (blob), directory (tree), and commit as a **hash of its content** (SHA-1/SHA-256).
- The hash IS the identifier: `{hash → content}`.
- If two files have the same content → same hash → stored only once (deduplication).
- Checking if a file changed: compare hashes in O(1) instead of comparing file contents.

```
File: "Hello, World!" → SHA-1 → "943a702d06f34599aee1f8da8ef9f7296031d699"

Git stores: objects/94/3a702d06f34599aee1f8da8ef9f7296031d699

Changed file: "Hello, World!!" → SHA-1 → "different_hash"
→ Different hash = file changed. No need to diff the contents.
```

**Content-Addressable Storage (CAS):** Also used by Docker (image layers), IPFS, and cloud backup systems.

### 🔍 Scenario 10: Feature Flags / A/B Testing — User Assignment

**Interviewer:** _"How do you deterministically assign users to A/B test groups without storing their assignment?"_

**Answer:**
- Hash the user ID: `group = hash(userId + experimentName) % numGroups`.
- Same user always gets the same group (deterministic).
- No need to store assignments in a database — the hash function IS the assignment.
- O(1) per lookup, zero storage.

```
hash("user123" + "button_color_test") % 2 = 0 → Group A (blue button)
hash("user456" + "button_color_test") % 2 = 1 → Group B (green button)

// Same user always gets the same group
// Adding a new experiment doesn't affect existing assignments
```

**Used by:** Google, Facebook, Netflix for A/B test bucketing.

### 🔍 Scenario 11: Real-Time Analytics — Count Distinct Users

**Interviewer:** _"Your website gets 10 million page views per day. How do you count unique visitors in real-time?"_

**Answer:**
- **Exact:** Use a hash set of user IDs. O(1) insert, O(1) lookup, O(n) space.
- **Approximate (for very large scale):** Use **HyperLogLog** — a probabilistic data structure based on hashing that estimates cardinality using O(1) space (12 KB for billions of elements!) with ~2% error.

```
// Exact approach (hash set):
uniqueUsers = HashSet()
for each pageView:
    uniqueUsers.add(pageView.userId)
count = uniqueUsers.size()

// HyperLogLog approach (Redis):
PFADD daily_visitors "user123"
PFADD daily_visitors "user456"
PFCOUNT daily_visitors → approximate count
```

**Real-world:** Redis supports HyperLogLog natively. Google Analytics uses similar techniques.

### 🔍 Scenario 12: WebSocket — Online User Tracking

**Interviewer:** _"In your chat application, you need to show which users are currently online. When a user connects/disconnects, update the status. How do you implement this?"_

**Answer:**
- Hash map: `{userId → connectionInfo}`.
- On connect: `onlineUsers[userId] = {socketId, lastActive}` → O(1).
- On disconnect: `onlineUsers.erase(userId)` → O(1).
- Check if online: `onlineUsers.count(userId)` → O(1).
- Get all online users: iterate the map → O(n).

```
onlineUsers = HashMap<UserId, SocketInfo>()

// WebSocket events:
onConnect(userId, socket):
    onlineUsers[userId] = {socket, timestamp}

onDisconnect(userId):
    onlineUsers.erase(userId)

isOnline(userId):
    return onlineUsers.containsKey(userId)  // O(1)
```

### 🔍 Scenario 13: Middleware — Request Deduplication / Caching API Responses

**Interviewer:** _"Your API endpoint is expensive to compute. How do you cache responses so identical requests don't recompute?"_

**Answer:**
- Hash the request parameters to create a cache key.
- Use a hash map as the cache: `{requestHash → response}`.
- Before processing: check cache → O(1).
- Use TTL (time to live) for cache invalidation.

```
cache = HashMap<String, {response, expiry}>()

function handleRequest(params):
    cacheKey = hash(serialize(params))
    
    if cacheKey in cache and cache[cacheKey].expiry > now():
        return cache[cacheKey].response  // Cache HIT — O(1)
    
    response = expensiveComputation(params)
    cache[cacheKey] = {response, expiry: now() + 300s}
    return response
```

**Real-world:** Redis, Memcached, and in-memory caches like Guava Cache (Java) / node-cache (Node.js) all use hash maps internally.

### 🔍 Scenario 14: File Integrity — Checksums & Content Verification

**Interviewer:** _"A user uploads a large file to your server. How do you verify the upload wasn't corrupted? How do you check if the same file was already uploaded?"_

**Answer:**
- Compute a hash (SHA-256 / MD5) of the file content → O(file_size).
- **Integrity:** Compare hash before and after upload. If they match → file is intact.
- **Deduplication:** Check if hash exists in your hash set → O(1). If it does, skip storing a duplicate.

```
// Upload flow:
clientHash = sha256(fileData)  // Computed on client
// ... upload ...
serverHash = sha256(receivedData)  // Computed on server

if clientHash != serverHash:
    return "Upload corrupted, retry"

// Deduplication:
if serverHash in storedFiles:
    return "File already exists, reusing"

storeFile(fileData)
storedFiles.add(serverHash)
```

**Used by:** Dropbox, Google Drive, AWS S3 (ETags are MD5 hashes of content).

### 🔍 Scenario 15: Bloom Filters — Membership Testing at Scale

**Interviewer:** _"You have a blocklist of 1 billion malicious URLs. Checking every web request against this list must be fast and memory-efficient. How?"_

**Answer:**
- A **Bloom Filter** is a probabilistic hash-based data structure.
- Uses k hash functions and a bit array.
- Insert: hash the URL k times, set those bits.
- Lookup: hash the URL k times, check if all bits are set.
- **False positives possible** (says "maybe in set"), **false negatives impossible** (if it says "not in set", it's definitely not).
- Space: ~1 byte per element for 1% false positive rate. 1 billion URLs ≈ 1 GB.

```
Bloom Filter (bit array of size m):
  Insert "badsite.com":
    hash1("badsite.com") % m = 42  → set bit 42
    hash2("badsite.com") % m = 108 → set bit 108
    hash3("badsite.com") % m = 230 → set bit 230
  
  Check "goodsite.com":
    hash1("goodsite.com") % m = 42  → bit set? ✅
    hash2("goodsite.com") % m = 55  → bit set? ❌
    → DEFINITELY NOT in blocklist

  Check "badsite.com":
    All 3 bits set → PROBABLY in blocklist (verify against full DB)
```

**Used by:** Google Chrome (safe browsing), Medium (recommendation dedup), Akamai (CDN cache).

| Question | Hashing | Alternative | When to Prefer Hashing |
|----------|---------|-------------|----------------------|
| _"Why not use an array?"_ | O(1), keys can be anything | O(1) but keys must be small ints | When keys are strings, large ints, or sparse |
| _"Why not use a BST / TreeMap?"_ | O(1) average, unordered | O(log n), sorted | When you don't need ordering or range queries |
