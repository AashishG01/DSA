## 1. What is the STL?

The **Standard Template Library (STL)** is C++'s built-in toolkit of **containers** (data structures), **algorithms** (sorting, searching, etc.), and **iterators** (pointers that traverse containers). It lets you write powerful, efficient code without reinventing the wheel.

### Why STL Matters for Interviews

- **Speed:** You can't afford to code a hash map from scratch during a 45-min interview.
- **Correctness:** STL implementations are battle-tested. Your hand-written ones have bugs.
- **Expression:** `sort(v.begin(), v.end())` vs writing 20 lines of Quick Sort.
- **Expected knowledge:** Interviewers expect fluency with STL. Not knowing it is a red flag.

### Core Philosophy

> **STL = Containers + Algorithms + Iterators**
>
> Containers **store** data. Algorithms **operate** on data. Iterators **connect** algorithms to containers.

```
              ┌─────────────┐
              │  ALGORITHMS  │  sort, find, count, transform...
              └──────┬───────┘
                     │ (iterators connect them)
              ┌──────┴───────┐
              │  CONTAINERS  │  vector, map, set, queue...
              └──────────────┘
```

## 2. The Big Picture — STL Components

### Container Categories at a Glance

| Category | Containers | Key Property |
|----------|-----------|-------------|
| **Sequence** | `vector`, `deque`, `list`, `forward_list`, `array` | Ordered by insertion position |
| **Associative (Ordered)** | `set`, `multiset`, `map`, `multimap` | Sorted by key (BST-based) |
| **Unordered (Hash)** | `unordered_set`, `unordered_multiset`, `unordered_map`, `unordered_multimap` | Hashed, O(1) average |
| **Adaptors** | `stack`, `queue`, `priority_queue` | Restricted interface on top of other containers |

### 🧠 Memorization Trick — "S-A-U-A" (Say "sah-wah")

- **S**equence — order of insertion matters
- **A**ssociative — auto-sorted by key
- **U**nordered — hashed, O(1) lookup
- **A**daptors — restricted views (LIFO, FIFO, priority)

## 3. How to Decide Which Container to Use

This is the **most important section** for interviews and competitive programming. Master this decision tree.

### 🌳 The Master Decision Tree

```
What do I need?
│
├── Just need a resizable array?
│   └── vector ✅ (default choice for everything)
│
├── Need O(1) lookup by key?
│   ├── Key → Value mapping?
│   │   ├── Need sorted keys?     → map
│   │   └── Don't need sorted?    → unordered_map ✅ (faster)
│   └── Just existence checking?
│       ├── Need sorted?          → set
│       └── Don't need sorted?    → unordered_set ✅ (faster)
│
├── Need to maintain sorted order automatically?
│   ├── Unique elements?          → set
│   ├── Allow duplicates?         → multiset
│   ├── Key-value with unique keys? → map
│   └── Key-value with dup keys?  → multimap
│
├── Need a stack (LIFO)?          → stack
├── Need a queue (FIFO)?          → queue
├── Need a priority queue (max/min)? → priority_queue
│
├── Need O(1) insert/delete at BOTH ends?
│   └── deque ✅
│
├── Need O(1) insert/delete in the MIDDLE?
│   └── list (doubly linked list)
│
├── Need fixed-size array?
│   └── array
│
└── Not sure?
    └── Start with vector. Change later if needed.
```

### 🧠 Memorization Rule — "When in Doubt, Use vector"

> `vector` is the correct choice **90% of the time**. Only switch when you have a specific need:
> - Need O(1) lookup → `unordered_map` / `unordered_set`
> - Need sorted data → `set` / `map`
> - Need LIFO → `stack`
> - Need FIFO → `queue`
> - Need max/min → `priority_queue`

### Quick Decision Table

| I need to... | Use this | Time |
|-------------|----------|------|
| Store elements in order | `vector` | O(1) push_back |
| Look up by key | `unordered_map` | O(1) avg |
| Check existence | `unordered_set` | O(1) avg |
| Keep elements sorted | `set` / `map` | O(log n) |
| Get min/max quickly | `priority_queue` | O(1) top, O(log n) push/pop |
| LIFO (last in, first out) | `stack` | O(1) |
| FIFO (first in, first out) | `queue` | O(1) |
| Count frequencies | `unordered_map<T, int>` | O(1) per update |
| Find kth smallest | `set` + iterator | O(log n) |
| Sliding window min/max | `deque` / `multiset` | O(1) / O(log n) |

## 4. Sequence Containers

### 4.1 vector — The King of Containers 👑

> **Use `vector` unless you have a specific reason not to.** It's the most cache-friendly, most flexible, and most commonly used container.

#### What It Is

A **dynamic array** that grows automatically. Contiguous memory, O(1) random access.

#### Complete API Reference

```cpp
#include <vector>

// ═══════════════════ CREATION ═══════════════════
vector<int> v;                        // empty
vector<int> v(5);                     // [0,0,0,0,0]
vector<int> v(5, 10);                // [10,10,10,10,10]
vector<int> v = {1, 2, 3};           // initializer list
vector<int> v(other.begin(), other.end()); // copy from range
vector<vector<int>> grid(m, vector<int>(n, 0)); // m×n matrix

// ═══════════════════ SIZE & CAPACITY ═══════════════════
v.size();         // number of elements (unsigned! be careful)
v.empty();        // true if size == 0
v.capacity();     // allocated capacity (≥ size)
v.resize(10);     // resize to 10 (pads with 0 or truncates)
v.resize(10, -1); // resize, fill new slots with -1
v.reserve(1000);  // pre-allocate (avoids reallocations)
v.shrink_to_fit(); // free unused capacity
v.clear();        // remove all elements (size = 0, capacity unchanged)

// ═══════════════════ ACCESS ═══════════════════
v[i];             // no bounds check — FAST but unsafe
v.at(i);          // bounds check — throws out_of_range
v.front();        // first element (undefined if empty!)
v.back();         // last element (undefined if empty!)
v.data();         // raw pointer to underlying array

// ═══════════════════ ADD / REMOVE ═══════════════════
v.push_back(10);      // add at end — O(1) amortized ✅
v.emplace_back(10);   // add at end (constructs in-place, avoids copy) ✅
v.pop_back();         // remove last — O(1) ✅
v.insert(v.begin() + i, val);  // insert at index i — O(n) ⚠️
v.erase(v.begin() + i);        // remove at index i — O(n) ⚠️
v.erase(v.begin() + l, v.begin() + r); // erase range [l, r)

// ═══════════════════ ITERATION ═══════════════════
for (int i = 0; i < v.size(); i++) { }    // index-based
for (int x : v) { }                        // range-based (copy)
for (int& x : v) { x *= 2; }              // range-based (reference, can modify)
for (const int& x : v) { }                // range-based (const reference)
for (auto it = v.begin(); it != v.end(); ++it) { } // iterator

// ═══════════════════ COMMON OPERATIONS ═══════════════════
sort(v.begin(), v.end());                  // sort ascending
sort(v.begin(), v.end(), greater<int>());  // sort descending
reverse(v.begin(), v.end());
*max_element(v.begin(), v.end());          // max value
*min_element(v.begin(), v.end());          // min value
accumulate(v.begin(), v.end(), 0);         // sum (include <numeric>)
count(v.begin(), v.end(), val);            // count occurrences
find(v.begin(), v.end(), val);             // returns iterator
binary_search(v.begin(), v.end(), val);    // true/false (sorted only)
lower_bound(v.begin(), v.end(), val);      // first >= val (sorted)
upper_bound(v.begin(), v.end(), val);      // first > val (sorted)
unique(v.begin(), v.end());                // remove adjacent dups (sorted first!)
```

#### 🧠 Memorization — vector Operations by Complexity

```
O(1):  push_back, pop_back, front, back, [i], size, empty
O(n):  insert (middle), erase (middle), find, count
O(n log n): sort
```

#### ⚠️ Common Pitfalls

| Pitfall | Why it's dangerous | Fix |
|---------|-------------------|-----|
| `v.size() - 1` when empty | `size()` returns unsigned; 0 - 1 = huge number | Check `!v.empty()` first, or cast to `int` |
| `for (int i = 0; i < v.size(); i++)` | Comparing signed `int` with unsigned `size_t` | Use `int n = v.size(); for (int i = 0; i < n; i++)` |
| Accessing `v[i]` out of bounds | No error, just undefined behavior | Use `v.at(i)` for safety in debugging |
| Iterating and modifying | Invalidates iterators | Collect indices first, or use index-based loop |
| Not using `reserve()` | Many reallocations when size is known | `v.reserve(expectedSize)` |

### 4.2 deque — Double-Ended Queue

> **Use when you need O(1) push/pop at BOTH front and back.**

```cpp
#include <deque>

deque<int> dq;
dq.push_back(10);     // add to back — O(1)
dq.push_front(20);    // add to front — O(1) ✅ (vector can't do this!)
dq.pop_back();         // remove from back — O(1)
dq.pop_front();        // remove from front — O(1) ✅
dq[i];                 // random access — O(1)
dq.front();            // first element
dq.back();             // last element
dq.size();
```

#### When to Use deque vs vector

| Need | vector | deque |
|------|--------|-------|
| Push/pop at back only | ✅ Faster (contiguous) | ✅ |
| Push/pop at front | ❌ O(n) | ✅ O(1) |
| Random access | ✅ O(1) | ✅ O(1) (slightly slower) |
| Cache performance | ✅ Excellent | ⚠️ Good but not contiguous |
| Memory | Less overhead ✅ | More overhead (block-based) |

#### 🧠 Memorization: "deque = vector that also works at the front"

**Interview use:** Sliding window maximum (monotonic deque), BFS with 0-1 weights.

---

### 4.3 list — Doubly Linked List

> **Use when you need O(1) insert/delete at ANY position (given an iterator).**

```cpp
#include <list>

list<int> lst;
lst.push_back(10);       // O(1)
lst.push_front(20);      // O(1)
lst.pop_back();           // O(1)
lst.pop_front();          // O(1)
auto it = lst.begin();
advance(it, 3);           // move iterator forward 3 positions — O(n)!
lst.insert(it, 99);       // insert before iterator — O(1)
lst.erase(it);            // remove at iterator — O(1)
lst.splice(pos, other_list); // move elements between lists — O(1)
lst.sort();               // built-in merge sort — O(n log n)
lst.merge(other_list);    // merge two sorted lists — O(n)
lst.reverse();            // O(n)
lst.unique();             // remove adjacent duplicates
```

#### When to Use list

- **Almost never in interviews.** `vector` + `erase()` is usually simpler and faster.
- **Use for LRU Cache:** `list` + `unordered_map<key, list::iterator>` → O(1) access + O(1) move to front.
- **Use when pointers/iterators must stay valid** after insert/erase (vector invalidates them).

#### 🧠 Memorization: "list = O(1) insert/delete anywhere, but NO random access"

---

### 4.4 forward_list — Singly Linked List

> **Rarely used.** Like `list` but only forward traversal. Saves memory (no backward pointer).

```cpp
#include <forward_list>
forward_list<int> fl = {1, 2, 3};
fl.push_front(0);         // O(1) — can only add at front
fl.insert_after(fl.begin(), 99); // insert after position
fl.erase_after(fl.begin());      // erase after position
```

---

### 4.5 array — Fixed-Size Array (C++11)

> **Use when size is known at compile time.**

```cpp
#include <array>
array<int, 5> a = {1, 2, 3, 4, 5};
a.size();    // 5 (constexpr)
a[0];        // access
a.at(0);     // bounds-checked access
a.fill(0);   // fill all with 0
a.front();   // first
a.back();    // last
sort(a.begin(), a.end()); // works with STL algorithms
```

**Advantage over C arrays:** Has `.size()`, works with STL algorithms, no pointer decay.

| Feature | `vector` | `deque` | `list` | `forward_list` | `array` |
|---------|----------|---------|--------|----------------|---------|
| Random access | O(1) ✅ | O(1) | ❌ O(n) | ❌ O(n) | O(1) ✅ |
| Push back | O(1)* | O(1)* | O(1) | ❌ | ❌ Fixed |

| Size | Dynamic | Dynamic | Dynamic | Dynamic | Fixed |

## 5. Associative Containers (Ordered)

> These use **self-balancing BSTs (Red-Black Trees)** internally. All operations are **O(log n)**. Elements are always **sorted by key**.

### 5.1 set — Sorted Unique Elements

```cpp
#include <set>

set<int> s;

// ═══════════════════ INSERT ═══════════════════
s.insert(10);         // O(log n) — returns {iterator, bool}
s.insert({1, 2, 3});  // insert multiple
s.emplace(10);        // construct in-place

// ═══════════════════ FIND / CHECK ═══════════════════
s.count(10);          // 0 or 1 (since unique) — O(log n)
s.find(10);           // returns iterator or s.end() — O(log n)
s.contains(10);       // C++20: true/false — O(log n)

// ═══════════════════ DELETE ═══════════════════
s.erase(10);          // erase by value — O(log n)
s.erase(s.begin());   // erase by iterator — O(1) amortized
s.clear();            // remove all

// ═══════════════════ ORDER OPERATIONS (THE POWER OF set) ═══════════════════
s.begin();            // iterator to SMALLEST element
s.rbegin();           // iterator to LARGEST element
*s.begin();           // smallest value
*s.rbegin();          // largest value

s.lower_bound(x);    // first element >= x — O(log n) ✅
s.upper_bound(x);    // first element > x  — O(log n) ✅

auto it = s.lower_bound(5);
if (it != s.end()) cout << *it;  // smallest element >= 5

// Previous element (predecessor)
auto it = s.upper_bound(x);
if (it != s.begin()) { --it; /* *it is largest element <= x */ }

// ═══════════════════ ITERATION (always sorted!) ═══════════════════
for (int x : s) cout << x << " ";  // prints in sorted order automatically
```

#### When to Use set

- Need **sorted unique elements** with O(log n) insert/find/delete.
- Need `lower_bound` / `upper_bound` (range queries).
- Need **min/max** at any time: `*s.begin()` / `*s.rbegin()`.
- Sliding window with ordered operations.

#### 🧠 Memorization: "set = sorted unique values, O(log n) everything, has lower_bound"

---

### 5.2 multiset — Sorted Elements with Duplicates

Same as `set` but **allows duplicate values**.

```cpp
#include <set>

multiset<int> ms;
ms.insert(5);
ms.insert(5);           // allowed! ms = {5, 5}
ms.count(5);             // returns 2 (actual count!)
ms.erase(5);             // ⚠️ removes ALL 5s!
ms.erase(ms.find(5));   // removes only ONE 5 ✅

// All set operations work: lower_bound, upper_bound, begin, rbegin
```

#### When to Use multiset

- **Sliding window with sorted access:** Track all elements in the window, get min/max, find kth element.
- **Maintaining sorted order with duplicates.**

#### 🧠 Memorization: "multiset = set that allows duplicates. Use erase(find(x)) to remove ONE."

---

### 5.3 map — Sorted Key-Value Pairs

```cpp
#include <map>

map<string, int> m;

// ═══════════════════ INSERT / UPDATE ═══════════════════
m["apple"] = 5;          // insert or update — O(log n)
m.insert({"banana", 3}); // insert only if key doesn't exist
m.emplace("cherry", 7);  // construct in-place
m["apple"]++;             // increment value — O(log n)

// ═══════════════════ ACCESS ═══════════════════
m["apple"];       // returns value (⚠️ creates entry with default value if key missing!)
m.at("apple");    // returns value (throws if key missing) ✅ safer

// ═══════════════════ FIND / CHECK ═══════════════════
m.count("apple");   // 0 or 1
m.find("apple");    // iterator or m.end()
if (m.find("apple") != m.end()) { /* exists */ }

// ═══════════════════ DELETE ═══════════════════
m.erase("apple");       // by key
m.erase(m.begin());     // by iterator

// ═══════════════════ ITERATION (sorted by key!) ═══════════════════
for (auto& [key, val] : m) {
    cout << key << ": " << val << endl;  // alphabetical order
}

// ═══════════════════ ORDER OPERATIONS ═══════════════════
m.begin()->first;         // smallest key
m.rbegin()->first;        // largest key
m.lower_bound("b");       // first key >= "b"
m.upper_bound("b");       // first key > "b"
```

#### ⚠️ The #1 map Pitfall

```cpp
// THIS CREATES AN ENTRY!
if (m["unknown_key"] == 0) { ... }
// Now "unknown_key" exists in the map with value 0!

// USE THIS INSTEAD:
if (m.find("unknown_key") != m.end()) { ... }
// Or:
if (m.count("unknown_key")) { ... }
```

#### 🧠 Memorization: "map = sorted dictionary. O(log n). Don't use [] to check existence!"

---

### 5.4 multimap — Sorted Key-Value with Duplicate Keys

```cpp
#include <map>

multimap<string, int> mm;
mm.insert({"a", 1});
mm.insert({"a", 2});  // allowed! Two entries with key "a"
// mm["a"] = 3;       // ❌ operator[] NOT available in multimap!

auto range = mm.equal_range("a"); // returns pair of iterators
for (auto it = range.first; it != range.second; it++)
    cout << it->second; // prints 1, 2
```

#### 🧠 Memorization: "multimap = map with duplicate keys. No [] operator. Use equal_range."

---

### 🔄 Ordered Container Comparison

| Feature | `set` | `multiset` | `map` | `multimap` |
|---------|-------|------------|-------|------------|
| Duplicates | ❌ | ✅ | ❌ (keys) | ✅ (keys) |
| Access | By value | By value | By key `[]` | By key (no `[]`) |
| Insert | O(log n) | O(log n) | O(log n) | O(log n) |
| Find | O(log n) | O(log n) | O(log n) | O(log n) |
| Delete | O(log n) | O(log n) | O(log n) | O(log n) |
| Sorted | ✅ | ✅ | ✅ by key | ✅ by key |
| lower/upper_bound | ✅ | ✅ | ✅ | ✅ |

## 6. Unordered Containers (Hash-Based)

> These use **hash tables** internally. Average **O(1)**, worst case O(n). **No ordering.**

### 6.1 unordered_set

```cpp
#include <unordered_set>

unordered_set<int> us;
us.insert(10);          // O(1) avg
us.count(10);           // 0 or 1 — O(1) avg
us.find(10);            // iterator or us.end() — O(1) avg
us.erase(10);           // O(1) avg
us.size();
us.empty();

for (int x : us) cout << x;  // ⚠️ NO guaranteed order!
```

### 6.2 unordered_map

```cpp
#include <unordered_map>

unordered_map<string, int> um;
um["apple"] = 5;          // O(1) avg
um.insert({"banana", 3}); // O(1) avg
um.count("apple");         // O(1) avg
um.find("apple");          // O(1) avg
um.erase("apple");         // O(1) avg
um["apple"]++;             // frequency counting pattern

for (auto& [key, val] : um) cout << key << ": " << val;  // random order!
```

### 6.3 unordered_multiset & unordered_multimap

Same as above but allow duplicates. Rarely used in interviews.

### Ordered vs Unordered — The Ultimate Comparison

| Feature | `map` / `set` | `unordered_map` / `unordered_set` |
|---------|--------------|----------------------------------|
| **Internal DS** | Red-Black Tree | Hash Table |
| **Insert/Find/Delete** | O(log n) guaranteed | O(1) average, O(n) worst |
| **Sorted iteration?** | ✅ Yes | ❌ No |
| **lower_bound/upper_bound?** | ✅ Yes | ❌ No |
| **Memory** | Less | More (hash table overhead) |
| **Custom key type** | Need `operator<` | Need hash function + `operator==` |
| **Cache performance** | Poor (tree pointer chasing) | Good (array-based) |
| **Worst case (adversarial)** | O(log n) always | O(n) — can be exploited! |

### 🧠 Memorization Rule — "Ordered vs Unordered"

> **Use `unordered_` (hash) by default** — it's faster (O(1) vs O(log n)).  
> **Switch to ordered (tree) ONLY when you need:**
> 1. Sorted iteration
> 2. `lower_bound` / `upper_bound`
> 3. Guaranteed worst-case performance
> 4. Min/max element queries

## 7. Container Adaptors

> Adaptors **wrap** an underlying container and **restrict** its interface. They don't have iterators or `begin()`/`end()`.

### 7.1 stack — Last In, First Out (LIFO)

```cpp
#include <stack>

stack<int> st;
st.push(10);     // O(1)
st.push(20);
st.top();        // 20 (peek, doesn't remove) — O(1)
st.pop();        // removes 20 (⚠️ returns void!) — O(1)
st.size();       // 1
st.empty();      // false
```

#### 🧠 Memorization: "Push, Top, Pop. Pop returns VOID!"

#### When to Use stack

- **Bracket matching:** `({[]})`
- **Monotonic stack:** Next greater/smaller element
- **DFS (iterative):** Replace recursion with explicit stack
- **Expression evaluation:** Infix/postfix conversion
- **Undo operations**

---

### 7.2 queue — First In, First Out (FIFO)

```cpp
#include <queue>

queue<int> q;
q.push(10);      // add to back — O(1)
q.push(20);
q.front();       // 10 (first element)
q.back();        // 20 (last element)
q.pop();         // removes 10 (front) — O(1)
q.size();
q.empty();
```

#### 🧠 Memorization: "Push (back), Front, Pop (front). Like a real queue/line."

#### When to Use queue

- **BFS (Breadth-First Search):** The classic use case
- **Level-order traversal:** Trees
- **Task scheduling:** FIFO processing
- **Sliding window** (sometimes, with deque)

---

### 7.3 priority_queue — Heap (Max by Default)

```cpp
#include <queue>

// ═══════════════════ MAX-HEAP (default) ═══════════════════
priority_queue<int> maxHeap;
maxHeap.push(10);     // O(log n)
maxHeap.push(30);
maxHeap.push(20);
maxHeap.top();        // 30 (maximum) — O(1)
maxHeap.pop();        // removes 30 — O(log n)

// ═══════════════════ MIN-HEAP ═══════════════════
priority_queue<int, vector<int>, greater<int>> minHeap;
minHeap.push(10);
minHeap.push(30);
minHeap.push(20);
minHeap.top();        // 10 (minimum) — O(1)

// ═══════════════════ CUSTOM COMPARATOR ═══════════════════
// Sort by second value (ascending)
auto cmp = [](pair<int,int>& a, pair<int,int>& b) {
    return a.second > b.second;  // ⚠️ "greater" for min-heap!
};
priority_queue<pair<int,int>, vector<pair<int,int>>, decltype(cmp)> pq(cmp);
```

#### ⚠️ Priority Queue Gotcha — Comparator is REVERSED!

```
greater<int> → MIN-heap  (opposite of what you might expect)
less<int>    → MAX-heap  (default)
```

> **Think of it as:** The comparator defines what goes to the BOTTOM. `greater<int>` puts greater values at the bottom → smallest on top → min-heap.

#### 🧠 Memorization: "PQ is a MAX-heap. For MIN-heap: add `vector<int>, greater<int>`"

#### When to Use priority_queue

- **Top K elements:** Maintain heap of size K
- **Dijkstra's algorithm:** Min-heap of {distance, node}
- **Merge K sorted lists:** Min-heap of K list heads
- **Task scheduling by priority**
- **Kth largest/smallest:** Max-heap of size K for kth largest
- **Median from stream:** Max-heap + Min-heap

#### priority_queue Operations Summary

| Operation | Time | Notes |
|-----------|------|-------|
| `push()` | O(log n) | Insert element |
| `top()` | O(1) | View max (or min) |
| `pop()` | O(log n) | Remove max (or min) |
| `size()` | O(1) | |
| `empty()` | O(1) | |
| Find arbitrary element | ❌ Not supported | Can't search/iterate |
| Decrease key | ❌ Not supported | Push duplicate, ignore old |

## 8. Iterators

### What Are Iterators?

Iterators are **generalized pointers** that let you traverse containers. They bridge containers and algorithms.

```cpp
vector<int> v = {10, 20, 30};
auto it = v.begin();  // points to 10
*it;                   // dereference: 10
++it;                  // advance: points to 20
it + 2;                // random access: points to 30 (vector only)
```

### Iterator Categories

| Category | Operations | Containers |
|----------|-----------|-----------|
| **Input** | Read, `++` | `istream_iterator` |
| **Output** | Write, `++` | `ostream_iterator` |
| **Forward** | Read/write, `++` | `forward_list`, `unordered_*` |
| **Bidirectional** | `++`, `--` | `list`, `set`, `map` |
| **Random Access** | `++`, `--`, `+n`, `-n`, `[]` | `vector`, `deque`, `array` |

### 🧠 Memorization: "Random > Bidirectional > Forward > Input/Output"

> `vector` has the most powerful iterators (random access). `set`/`map` have bidirectional. `unordered_*` have forward only.

### Common Iterator Patterns

```cpp
// begin/end
auto it = v.begin();  // first element
auto it = v.end();    // ONE PAST last element (don't dereference!)

// reverse
auto it = v.rbegin(); // last element
auto it = v.rend();   // one before first

// advance (works with all iterators)
auto it = s.begin();
advance(it, 3);       // move forward 3 positions

// distance
int d = distance(v.begin(), it);  // how many steps from begin to it

// next / prev (C++11)
auto it2 = next(it);    // next element
auto it3 = prev(it);    // previous element
auto it4 = next(it, 5); // 5 elements forward
```

### ⚠️ Iterator Invalidation Rules

| Container | Operations that invalidate iterators |
|-----------|--------------------------------------|
| `vector` | `push_back` (if reallocation), `insert`, `erase` |
| `deque` | `push_front`, `push_back`, `insert`, `erase` |
| `list` | Only the erased element's iterator |
| `set/map` | Only the erased element's iterator |
| `unordered_*` | Rehashing (on insert if load factor exceeded) |

> **Rule of thumb:** `list`, `set`, `map` are safe — only erased iterators are invalidated. `vector` and `deque` are dangerous — many operations invalidate iterators.

### 9.1 Sorting Algorithms

```cpp
#include <algorithm>

vector<int> v = {5, 3, 8, 1, 2};

// ═══════════════════ BASIC SORT ═══════════════════
sort(v.begin(), v.end());                    // ascending [1,2,3,5,8]
sort(v.begin(), v.end(), greater<int>());    // descending [8,5,3,2,1]

// Custom comparator
sort(v.begin(), v.end(), [](int a, int b) {
    return abs(a) < abs(b);  // sort by absolute value
});

// ═══════════════════ STABLE SORT ═══════════════════
stable_sort(v.begin(), v.end());  // preserves relative order of equal elements

// ═══════════════════ PARTIAL SORT ═══════════════════
partial_sort(v.begin(), v.begin() + 3, v.end());
// First 3 elements are the 3 smallest, sorted. Rest: unspecified.
// Use when you only need top K sorted.

// ═══════════════════ NTH ELEMENT ═══════════════════
nth_element(v.begin(), v.begin() + k, v.end());
// v[k] is the element that WOULD BE at index k if sorted.
// Elements before k are ≤ v[k], after are ≥ v[k]. NOT sorted.
// Average O(n)! Use for finding median, kth element.

// ═══════════════════ IS SORTED ═══════════════════
is_sorted(v.begin(), v.end());  // true if sorted ascending
```

#### 🧠 When to Use Which Sort

| Need | Use | Time |
|------|-----|------|
| Full sort | `sort` | O(n log n) |
| Stable full sort | `stable_sort` | O(n log n) |
| Only top K sorted | `partial_sort` | O(n log K) |
| Only kth element | `nth_element` | O(n) avg |
| Check if sorted | `is_sorted` | O(n) |

### 9.2 Searching Algorithms

```cpp
// ═══════════════════ LINEAR SEARCH ═══════════════════
auto it = find(v.begin(), v.end(), target);    // O(n)
if (it != v.end()) cout << "Found at index " << (it - v.begin());

// Find with condition
auto it = find_if(v.begin(), v.end(), [](int x) { return x > 5; });
// First element > 5

auto it = find_if_not(v.begin(), v.end(), [](int x) { return x % 2 == 0; });
// First odd element

// ═══════════════════ BINARY SEARCH (sorted only!) ═══════════════════
bool found = binary_search(v.begin(), v.end(), target);  // true/false

auto lb = lower_bound(v.begin(), v.end(), target);  // first >= target
auto ub = upper_bound(v.begin(), v.end(), target);  // first > target

// Number of occurrences of target in sorted array:
int count = upper_bound(v.begin(), v.end(), target) - lower_bound(v.begin(), v.end(), target);

// Equal range: returns {lower_bound, upper_bound} as a pair
auto [lo, hi] = equal_range(v.begin(), v.end(), target);
```

#### 🧠 lower_bound vs upper_bound — The Ultimate Guide

```
Sorted array: [1, 2, 2, 2, 3, 5, 8]
                       ↑        ↑
              lower_bound(2)  upper_bound(2)
              (first >= 2)    (first > 2)

lower_bound(2) → points to index 1 (first 2)
upper_bound(2) → points to index 4 (first element after all 2s = 3)
count of 2s = upper_bound - lower_bound = 4 - 1 = 3 ✅

lower_bound(4) → points to index 5 (value 5, first >= 4)
upper_bound(4) → points to index 5 (value 5, first > 4)
```

**Memorization:**
- `lower_bound` = "where would I INSERT this value to keep it sorted?" (first valid position)
- `upper_bound` = "where is the LAST valid insert position?"

### 9.4 Modifying Algorithms

```cpp
// ═══════════════════ FILL / GENERATE ═══════════════════
fill(v.begin(), v.end(), 0);         // set all to 0
fill_n(v.begin(), 5, -1);           // set first 5 to -1
iota(v.begin(), v.end(), 1);         // fill with 1,2,3,4,5... (include <numeric>)
generate(v.begin(), v.end(), []() { return rand() % 100; }); // random fill

// ═══════════════════ TRANSFORM ═══════════════════
transform(v.begin(), v.end(), v.begin(), [](int x) { return x * 2; });
// Double every element in-place

// Transform and store in another container
vector<int> result(v.size());
transform(v.begin(), v.end(), result.begin(), [](int x) { return x * x; });

// ═══════════════════ REPLACE ═══════════════════
replace(v.begin(), v.end(), 0, -1);  // replace all 0s with -1
replace_if(v.begin(), v.end(), [](int x) { return x < 0; }, 0); // negatives → 0

// ═══════════════════ REMOVE (tricky!) ═══════════════════
// remove does NOT actually remove! It moves unwanted elements to the end
// and returns an iterator to the new logical end. You must erase the rest.

// The ERASE-REMOVE idiom:
v.erase(remove(v.begin(), v.end(), 5), v.end());  // remove all 5s ✅
v.erase(remove_if(v.begin(), v.end(), [](int x) { return x < 0; }), v.end()); // remove negatives

// ═══════════════════ UNIQUE ═══════════════════
sort(v.begin(), v.end());  // MUST sort first!
v.erase(unique(v.begin(), v.end()), v.end());  // remove duplicates

// ═══════════════════ REVERSE / ROTATE ═══════════════════
reverse(v.begin(), v.end());

// Rotate left by 2 positions: [1,2,3,4,5] → [3,4,5,1,2]
rotate(v.begin(), v.begin() + 2, v.end());

// ═══════════════════ SWAP ═══════════════════
swap(a, b);                              // swap two values
swap_ranges(v1.begin(), v1.end(), v2.begin()); // swap two ranges
iter_swap(it1, it2);                     // swap values at two iterators
```

#### 🧠 The Erase-Remove Idiom — MUST KNOW

```
v.erase(remove(v.begin(), v.end(), val), v.end());
       ^^^^^^                              ^^^^^
       moves val to end,                   erases from
       returns new end                     new end to old end
```

> **MEMORIZE THIS LINE.** It's asked in interviews and used in every codebase.

## 11. Strings

### string — Complete Reference

```cpp
#include <string>

// ═══════════════════ CREATION ═══════════════════
string s = "hello";
string s(5, 'a');           // "aaaaa"
string s = to_string(42);  // "42" (int → string)
int n = stoi("42");         // 42 (string → int)
long long n = stoll("123456789"); // string → long long
double d = stod("3.14");   // string → double

// ═══════════════════ ACCESS ═══════════════════
s[0];         // 'h' — no bounds check
s.at(0);      // 'h' — bounds check
s.front();    // 'h'
s.back();     // 'o'
s.length();   // 5 (same as s.size())
s.empty();    // false

// ═══════════════════ MODIFICATION ═══════════════════
s += " world";              // append string
s.push_back('!');           // append char
s.pop_back();               // remove last char
s.append("!!!");            // append
s.insert(5, " beautiful");  // insert at position 5
s.erase(5, 10);             // erase 10 chars starting at position 5
s.replace(0, 5, "HI");     // replace first 5 chars with "HI"
s.clear();                  // empty the string

// ═══════════════════ SEARCH ═══════════════════
size_t pos = s.find("llo");         // 2 (first occurrence)
size_t pos = s.rfind("l");          // 3 (last occurrence)
size_t pos = s.find_first_of("aeiou"); // first vowel position
size_t pos = s.find_last_of("aeiou");  // last vowel position
// Returns string::npos if not found
if (s.find("xyz") == string::npos) cout << "Not found";

// ═══════════════════ SUBSTRING ═══════════════════
string sub = s.substr(1, 3);   // 3 chars starting at index 1: "ell"
string sub = s.substr(2);      // from index 2 to end: "llo"

// ═══════════════════ COMPARE ═══════════════════
s == "hello";       // true
s < "world";        // true (lexicographic)
s.compare("hello"); // 0 (equal)

// ═══════════════════ CONVERSION ═══════════════════
// char → string
string s(1, 'a');              // "a"
string s = string() + 'a';    // "a"

// String of digits → number
int n = stoi("42");
int n = s[i] - '0';           // single char digit → int

// Number → string
string s = to_string(42);

// To uppercase/lowercase
transform(s.begin(), s.end(), s.begin(), ::toupper);
transform(s.begin(), s.end(), s.begin(), ::tolower);

// ═══════════════════ USEFUL PATTERNS ═══════════════════
// Reverse a string
reverse(s.begin(), s.end());

// Check palindrome
bool isPalin = (s == string(s.rbegin(), s.rend()));

// Split string by delimiter (no built-in split!)
stringstream ss("hello world foo");
string token;
vector<string> tokens;
while (ss >> token) tokens.push_back(token);  // split by whitespace

// Split by specific delimiter
stringstream ss("a,b,c,d");
while (getline(ss, token, ',')) tokens.push_back(token);  // split by comma

// Count character frequency
int freq[26] = {0};
for (char c : s) freq[c - 'a']++;

// Sort a string
sort(s.begin(), s.end());
```

### 12.3 Bit Manipulation Utilities

```cpp
__builtin_popcount(n);       // number of 1-bits (int)
__builtin_popcountll(n);     // for long long

// Check if power of 2
bool isPow2 = n > 0 && (n & (n - 1)) == 0;

// Lowest set bit
int lowest = n & (-n);

// Remove lowest set bit
n = n & (n - 1);
```

## 14. Interview Tips & Frequently Asked Concepts

### 💡 Top Interview Tips

1. **Write "using namespace std;" in interviews** — saves typing, accepted in competitive programming.
2. **Start with `vector` + `unordered_map`** — these two solve 70% of problems.
3. **Know time complexity of every operation you use** — interviewers will ask.
4. **`emplace_back` > `push_back`** — avoids copy construction.
5. **`reserve()` for known sizes** — avoids reallocation.
6. **Use structured bindings** — `auto [key, val] = *it;` is clean.

### 🎯 Frequently Asked Interview Questions

#### Q1: What's the difference between map and unordered_map?

**A:** `map` uses a Red-Black Tree (O(log n), sorted, has `lower_bound`). `unordered_map` uses a Hash Table (O(1) average, unsorted, no `lower_bound`). Use `unordered_map` unless you need sorted keys.

#### Q2: How does vector grow? What's the amortized complexity?

**A:** When `size == capacity`, vector allocates 2x capacity, copies all elements, frees old memory. Individual `push_back` is O(n) during reallocation, but total cost across n operations is O(n) → amortized O(1) per insertion.

#### Q3: When would you use set over unordered_set?

**A:** When you need: (1) sorted iteration, (2) `lower_bound`/`upper_bound`, (3) guaranteed O(log n) worst case (no hash collision attacks), (4) min/max via `begin()`/`rbegin()`.

#### Q4: What is iterator invalidation?

**A:** When container operations (insert, erase, reallocation) make existing iterators point to invalid memory. `vector` is most dangerous (reallocation invalidates ALL iterators). `set`/`map` only invalidate erased iterators. Always check the rules.

#### Q5: How would you implement an LRU Cache?

**A:** `list<pair<key, value>>` + `unordered_map<key, list::iterator>`.
- **Get:** Find in map → O(1). Move to front of list → O(1).
- **Put:** If full, remove `list.back()` (LRU). Insert at front. Update map. All O(1).

```cpp
class LRUCache {
    int capacity;
    list<pair<int,int>> cache;  // front = most recent
    unordered_map<int, list<pair<int,int>>::iterator> map;
public:
    int get(int key) {
        if (map.find(key) == map.end()) return -1;
        cache.splice(cache.begin(), cache, map[key]); // move to front
        return map[key]->second;
    }
    void put(int key, int value) {
        if (map.find(key) != map.end()) {
            map[key]->second = value;
            cache.splice(cache.begin(), cache, map[key]);
            return;
        }
        if (cache.size() == capacity) {
            map.erase(cache.back().first);
            cache.pop_back();
        }
        cache.push_front({key, value});
        map[key] = cache.begin();
    }
};
```

#### Q6: What does std::sort use internally?

**A:** **Introsort** = Quick Sort (fast average) + Heap Sort (avoids O(n²) worst case) + Insertion Sort (small partitions < 16). Guarantees O(n log n). Not stable.

#### Q7: How to use a custom type as a key in map/set vs unordered_map/set?

**A:**
- `map`/`set`: Define `operator<` or pass a custom comparator.
- `unordered_map`/`unordered_set`: Define a hash function + `operator==`.

```cpp
// For map/set:
struct Point {
    int x, y;
    bool operator<(const Point& other) const {
        return tie(x, y) < tie(other.x, other.y);
    }
};
set<Point> s;

// For unordered_map/set:
struct PointHash {
    size_t operator()(const Point& p) const {
        return hash<int>()(p.x) ^ (hash<int>()(p.y) << 32);
    }
};
unordered_set<Point, PointHash> us;
```

#### Q8: What's the difference between push_back and emplace_back?

**A:** `push_back(obj)` copies/moves `obj` into the container. `emplace_back(args...)` constructs the object **in-place** inside the container using the arguments, avoiding an extra copy/move. Prefer `emplace_back` for complex objects.

#### Q9: Why use reserve() on vectors?

**A:** Without `reserve()`, each time the vector runs out of capacity, it allocates 2x space and copies everything — O(n) per reallocation. With `reserve(n)`, you pre-allocate once. If you know you'll insert 1M elements, `v.reserve(1000000)` avoids ~20 reallocations.

#### Q10: What's the difference between lower_bound and upper_bound?

**A:** For sorted range with value `x`:
- `lower_bound(x)` → first element **≥ x** (insert position for x, before existing x's)
- `upper_bound(x)` → first element **> x** (insert position for x, after existing x's)
- Count of x = `upper_bound(x) - lower_bound(x)`

## 15. STL in Software Development & Web Development — Real-World Scenarios

> In SDE/web-dev interviews, the interviewer describes a **real product situation** and asks: _"Which data structure would you use? Why?"_ This section maps STL containers to real-world systems.

---

### 🔍 Scenario 1: Autocomplete — Type-Ahead Search

**Interviewer:** _"The user types 'ap' and sees suggestions: 'apple', 'application', 'appetizer'. How do you implement this?"_

**Answer:**
- Store all words in a **`set<string>`** (sorted).
- Use `lower_bound("ap")` to find the first word starting with "ap".
- Iterate forward until prefix no longer matches → O(log n + k results).
- Alternatively: `map<string, int>` for prefix → frequency, return top K.

```cpp
set<string> words = {"apple", "application", "appetizer", "banana"};
auto it = words.lower_bound("ap");
while (it != words.end() && it->substr(0, 2) == "ap") {
    cout << *it << endl;  // apple, appetizer, application
    ++it;
}
```

---

### 🔍 Scenario 2: Rate Limiter — Sliding Window Counter

**Interviewer:** _"Your API allows 100 requests per user per minute. How do you enforce this?"_

**Answer:**
- For each user: `unordered_map<string, queue<timestamp>>`.
- On each request: pop timestamps older than 60 seconds from front of `queue`.
- If `queue.size() < 100`, allow and push current timestamp. Else reject.
- All operations: O(1) amortized.

---

### 🔍 Scenario 3: Task Scheduler — Priority Processing

**Interviewer:** _"Your system processes tasks with priorities. Higher priority first. How?"_

**Answer:**
- Use `priority_queue<Task>` (max-heap by priority).
- `push()` new tasks → O(log n).
- `top()` gives highest priority → O(1).
- `pop()` removes it → O(log n).
- For priority + FIFO within same priority: `priority_queue<pair<int, int>>` where second = timestamp.

---

### 🔍 Scenario 4: Spell Checker — Dictionary Lookup

**Interviewer:** _"Your text editor checks if each word exists in a 100K-word dictionary. How?"_

**Answer:**
- Load dictionary into `unordered_set<string>`.
- Check each word: `dict.count(word)` → O(1) average.
- Total: O(n) for n words in the document.
- Why not `set`? No need for sorting, `unordered_set` is 5-10x faster for pure lookups.

---

### 🔍 Scenario 5: Browser History — Back/Forward Navigation

**Interviewer:** _"Implement browser back/forward buttons."_

**Answer:**
- Two `stack<string>` objects: `backStack` and `forwardStack`.
- **Navigate:** Push current URL to `backStack`, clear `forwardStack`, load new URL.
- **Back:** Push current to `forwardStack`, pop from `backStack`.
- **Forward:** Push current to `backStack`, pop from `forwardStack`.
- All operations: O(1).

---

### 🔍 Scenario 6: Event System — Publisher/Subscriber

**Interviewer:** _"Your app has events like 'user_login', 'order_placed'. Multiple handlers subscribe to each event. How do you store this?"_

**Answer:**
- `unordered_map<string, vector<function<void()>>>` — event name → list of handlers.
- Subscribe: `handlers["user_login"].push_back(callback)`.
- Publish: iterate over `handlers["user_login"]` and call each.
- O(1) lookup + O(k) for k handlers.

---

### 🔍 Scenario 7: Inventory System — Sorted Price Lookups

**Interviewer:** _"Your e-commerce system needs to find all products in a price range [low, high]. How?"_

**Answer:**
- Store `(price, product_id)` in a `set<pair<int, int>>` or `map<int, vector<Product>>`.
- Range query: iterate from `lower_bound(low)` to `upper_bound(high)`.
- O(log n + k) where k = number of results.
- Why `set` over `unordered_set`? Need range queries, which hash tables can't do.

---

### 🔍 Scenario 8: Music Player — Playlist Queue

**Interviewer:** _"Implement a music playlist with play next, add to queue, and shuffle."_

**Answer:**
- `deque<Song>` for the playlist.
- **Play next:** `pop_front()` → O(1).
- **Add to queue:** `push_back(song)` → O(1).
- **Play next (priority):** `push_front(song)` → O(1).
- **Shuffle:** `random_shuffle(playlist.begin(), playlist.end())`.

---

### 🔍 Scenario 9: Log Deduplication — Unique Recent Errors

**Interviewer:** _"Your monitoring system receives thousands of error logs. Show only unique errors from the last hour."_

**Answer:**
- `unordered_set<string>` for seen error signatures.
- `queue<pair<timestamp, string>>` for time-ordered tracking.
- On new error: if not in set, add to set and output. Push to queue.
- Periodically: pop from queue if timestamp > 1 hour, remove from set.

---

### 🔍 Scenario 10: Graph Algorithms — BFS/DFS/Dijkstra

**Interviewer:** _"Which STL containers for BFS? DFS? Dijkstra?"_

**Answer:**

| Algorithm | Container | Why |
|-----------|-----------|-----|
| BFS | `queue<int>` | FIFO exploration |
| DFS | `stack<int>` (iterative) | LIFO exploration |
| Dijkstra | `priority_queue<pair<int,int>, vector<...>, greater<...>>` | Min-heap for shortest distance |
| Adjacency list | `vector<vector<int>>` or `vector<vector<pair<int,int>>>` | (node, weight) pairs |
| Visited | `vector<bool>` or `unordered_set<int>` | Track visited nodes |

---

### 🔍 Scenario 11: Config Manager — Ordered Key-Value Store

**Interviewer:** _"Your app reads config from a file. You need to look up values by key and iterate in alphabetical order for display. Which container?"_

**Answer:**
- `map<string, string>` — O(log n) lookup, sorted iteration.
- NOT `unordered_map` because you need alphabetical display order.
- For reading: `map["database.host"] = "localhost"`.
- For display: `for (auto& [key, val] : config) { ... }` — automatically alphabetical.

---

### 🔍 Scenario 12: Real-Time Analytics — Top K Trending Items

**Interviewer:** _"Your app tracks click events. Show the top 5 most clicked items in the last hour."_

**Answer:**
- `unordered_map<string, int>` for frequency counting.
- `priority_queue<pair<int, string>, vector<...>, greater<...>>` (min-heap of size 5).
- For each item: increment count in map. Maintain heap of top 5.
- Push to heap → if size > 5, pop smallest → O(n log K), K = 5.
- Or: **Bucket sort approach** — `vector<vector<string>> buckets(maxFreq + 1)` → O(n).
